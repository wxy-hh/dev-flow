# 需求覆盖检查

本技能只回答“需求是否被计划和验证覆盖”，不评价架构优劣，也不写业务代码。

## 输入

读取用户给出的需求/设计、`work.md` 的 Boundary 和 Batches、必要源码事实。输入不足时说明缺什么，不猜测通过。

## 检查

为每个验收结果确认：

```text
结果 -> 来源 -> 对应批次 -> 验证 -> 状态
```

状态只使用：

- `COVERED`：有实现批次和可执行验证。
- `PARTIAL`：只覆盖部分分支或验证不完整。
- `MISSING`：没有对应实现或验证。
- `CONFLICT`：输入材料互相冲突。
- `OUT_OF_SCOPE`：计划触碰明确不做范围。

同时找没有需求来源的批次；必要的技术支撑批次写清理由即可。

## 输出

默认在对话中给出缺口优先的短表。存在 `work.md` 时，把已确认的修订直接合并到 Boundary/Batches，或追加一个简短 `Coverage` 小节；不另建报告。

`MISSING`、`CONFLICT` 或高后果的 `PARTIAL` 必须先解决或由用户明确接受，才能进入受影响实现。没有缺口时只写一句通过结论，不触发下一套固定流程。
