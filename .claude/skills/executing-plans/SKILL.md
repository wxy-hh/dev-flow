# 执行计划

## 开始前

1. 读取项目适配层、用户指定计划和相关源码。
2. 有 `work.md` 时只读取 Boundary、未完成 Batches、相关 Context、Risk and rollback 和当前 diff。
3. 校验计划仍适用于当前代码；发现明显错误先修订计划。
4. `Risks` 不是 `none` 时，只有 `Approval: confirmed`，或带明确证据的 `skipped`，才可写业务代码。

## 批次循环

每轮只处理一个未完成批次：

1. 确认 Outcome、Scope、依赖、验证和回撤。
2. 做满足 Outcome 的最小改动，不顺手扩范围。
3. 运行本批最相关验证并读取退出码/行为结果。
4. 验证通过后勾选批次，在 `Verification` 追加简短证据并更新 State。
5. 验证失败时先判断代码、环境或命令问题；修复后重跑受影响检查。不能证明完成时保持未勾选并标记 `Phase: blocked`。
6. 没有新风险、明显扩范围或外部授权需求时，自动进入下一批，不逐批询问用户。

不要等到所有代码写完后才更新恢复点。长任务在每个批次结束时都应能从 `work.md` 和当前 diff 恢复。

## 偏差处理

- 普通实现细节变化：自行修订当前批次说明并继续。
- 新增跨层传播、共享协议或协调回滚：重新判断等级，必要时补拆批次。
- 出现新风险标签或风险明显升级：更新 `Risks` 与控制，输出新的风险卡并停止。
- 用户要求跳过验证或接受失败：记录具体残留风险；未明确接受前不标记完成。

## 完成

所有批次完成后：

1. 基于完整 diff 做一次整体审查。
2. 运行项目适配层要求的最强相关验证和必要行为验证。
3. 更新 `Phase: done`、`Verification` 和最终 `Evidence`。
4. 有 `work.md` 时运行 feature checker。

不默认提交、推送、合并或生成新的流程文件。
