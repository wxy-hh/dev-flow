# 需求结构化澄清

目标是用最少交互得到可实现、可验收的边界，不是固定生成需求文档。

## 1. 先自行调查

读取项目适配层和直接相关源码，先确认能从仓库回答的事实：现有行为、接口、状态、相似实现、验证能力和限制。不要把可查问题抛给用户。

## 2. 只问阻塞问题

每轮一个问题。问题必须会实质改变以下至少一项：

- 用户可观察行为。
- 输入输出或错误分支。
- XS/S/M/L 判定。
- 风险标签或门禁。
- 验收和验证方式。

提供推荐默认值及理由，但允许用户自由回答。非阻塞细节使用现有项目模式或合理默认值继续。

## 3. 收敛边界

澄清后给出简短结果：

- Goal。
- In scope / Out of scope。
- 关键行为和异常分支。
- Acceptance。
- 已知约束与未决风险。
- 建议等级和风险标签。

存在 `work.md` 时更新其 Boundary/Risk；没有时留在对话。只有用户明确要求独立规格文档时才创建，并把它作为 `work.md` Context 的一个输入，不复制全文。

## 4. 返回主流程

req-probe 不自动写计划或代码。边界清楚后返回 dev-flow：普通任务选择最小路径；风险任务由 dev-flow 输出一次实现前风险卡。
