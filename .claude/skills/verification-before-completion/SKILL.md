# 完成前验证

核心规则：没有本轮实际证据，就不能声称完成。

## 准备

1. 读取 `.claude/rules/project-workflow.md` 的验证能力。
2. 查看当前 diff、Boundary 和风险标签。
3. 有 `work.md` 时读取未验证的批次、Verification 和 Accepted risks。
4. 选择能证明本次行为的最小验证集合，不机械运行所有命令。

## 验证层次

- 文档/规则：文本一致性、链接、脚本语法和相关自检。
- 类型/逻辑：相关 type-check、lint、测试或可执行小验证。
- 构建/接口：build、集成测试或真实请求路径。
- UI/运行时：浏览器自动化或可复查人工行为步骤。
- 高风险：成功、失败、缺失输入、权限/状态保护、清理、计费/用量和回撤后路径。

L 的运行时行为变化不能只靠 type-check 或 lint。没有真实测试运行器时如实说明，不把 dev/watch 命令当测试。

## 执行

每项记录：命令/操作、时间、退出码或实测结果。失败时区分：

- 代码问题：修复后重跑受影响验证。
- 命令不适用：换成能证明结论的检查。
- 环境阻塞：记录已检查内容和剩余风险，不伪造通过。

验证后若相关代码变化，旧证据失效。

## 结果

- `passed`：相关自动和行为验证均支持完成结论。
- `partial`：有明确缺口，且用户已接受具体残留风险。
- `failed`：存在失败或无法接受的缺口。

存在 `work.md` 时更新 State 的 Phase/Verification/Updated，并在 `## Verification` 写 `- Evidence: ...`。只有用户或外部环境必须独立执行步骤时才创建同目录 `manual-test.md`；执行结果仍回填工作文件。

没有工作文件时直接在最终回复报告证据。不要为轻量任务补建流程资产。
