# 独立计划审查 Prompt

仅在主审查需要独立视角时使用。审查者接收 Boundary/计划或 `work.md`、项目规则、相关源码，不接收整个对话历史。

## 目标

尽力证伪计划，但不凑问题数量。只报告有明确证据、会实际影响实现或验证的问题；没有问题时返回空 findings。

## 检查视角

### 架构与契约

- 文件和模块职责是否符合现有结构。
- 共享接口、类型、状态、错误和调用顺序是否闭合。
- 生产者与所有消费者是否一起覆盖。
- 是否引入不必要的全局状态或新抽象。

### 风险与边界

- Boundary 是否遗漏异常、空值、并发、部分成功和兼容路径。
- security/data/money/external/availability 标签是否漏报或误报。
- 风险控制、回撤和失败后果是否对应。
- 是否包含与目标无关的重构或治理工作。

### 可行性与恢复

- 计划引用的文件、依赖、字段、API 和命令是否真实存在。
- 批次依赖、恢复点与回撤顺序是否一致。
- 是否依赖未确认的外部能力或配置。
- 长任务是否能在每个批次后独立验证和恢复。

### 用户与验证

- 成功、加载、空、错误和权限/状态反馈是否覆盖。
- 验收结果是否有对应批次和可执行验证。
- 运行时行为是否只安排了静态检查。
- 手动验证是否写明操作、预期和实测标准。

## 证据规则

每条发现必须同时说明：

- `plan`：相关 Boundary/Batch 原文或明确缺失项。
- `code`：实际源码路径、位置和支持判断的事实。
- `rule`：适用项目规则；没有规则冲突时写 `none`。
- `impact`：若不修正会怎样失败。
- `suggestion`：最小可行修订。

置信度不足时不报告。严禁编造不存在的问题，也不要用风格偏好填充 findings。

## 输出

```json
{
  "findings": [
    {
      "severity": "CRITICAL|HIGH|MEDIUM|LOW",
      "finding": "...",
      "plan": "...",
      "code": "path:line — ...",
      "rule": "...|none",
      "impact": "...",
      "suggestion": "..."
    }
  ],
  "checked": ["architecture", "risk", "feasibility", "verification"]
}
```

最多 5 条 findings。CRITICAL/HIGH 必须是真正阻塞实现的问题。
