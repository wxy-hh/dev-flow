# 计划审查

计划审查发生在实现前，只检查计划是否足以安全执行；它不是每个 M/L 任务的固定仪式。

## 输入

优先读取用户指定的设计/计划、`work.md`、直接相关源码、项目适配层和命中的局部规则。不要只凭对话记忆，也不要加载无关仓库内容。

## 审查维度

1. Boundary：目标、非目标和验收是否一致。
2. Architecture：文件/模块职责与现有结构是否匹配。
3. Contract：共享类型、接口、状态、错误与调用顺序是否闭合。
4. Risk：标签是否漏报，控制是否覆盖真实失败后果。
5. Batches：依赖、部分成功、恢复点和回撤顺序是否合理。
6. Verification：每批和整体是否有能证明行为的检查。
7. Scope：是否加入与需求无关的重构或治理资产。

必要时使用 `references/review-prompts.md` 获取独立视角，但不要求凑够问题数量。

## 严重度

- CRITICAL：会导致安全/数据事故或计划无法成立。
- HIGH：明确的关键遗漏、契约错误、不可恢复顺序或主要链路无验证。
- MEDIUM：重要边界或验证不足，但可在实现中修正。
- LOW：与本次目标直接相关的可维护性建议。

每条问题必须包含证据、后果和最小修订。没有可靠问题时明确写“未发现阻塞项”。

## 输出与阻塞

默认在对话中输出 findings-first 结论。存在 `work.md` 时，把采纳的修订直接写回相应 Boundary、Batch、Risk 或 Verification，并可追加简短 `Plan review` 小节；不另建报告。

CRITICAL/HIGH 必须修正、被证据反驳，或由用户明确接受风险。审查发现新风险标签时返回 dev-flow 风险卡；否则审查结束即返回执行方，不自动串联其它技能。
