# 测试反模式

## 1. 只测试 mock

坏味道：

```typescript
expect(mockFn).toHaveBeenCalled()
```

这只能说明 mock 被调用，不说明真实行为正确。优先验证用户可观察结果、状态变化、返回值或 DOM 输出。

## 2. 测试名称太虚

坏：

```typescript
test('works')
```

好：

```typescript
test('保存失败时保留表单输入并显示错误信息')
```

## 3. 一个测试覆盖太多行为

如果测试名称里有多个“并且”，通常应该拆开。一个失败应当指向一个清晰问题。

## 4. 先写实现再补测试

补测很容易只证明当前实现存在，而不是证明需求正确。修 bug 时尤其要先写能复现 bug 的失败测试。

## 5. 断言实现细节

不要因为内部函数被调用就认为功能正确。实现可以重构，行为合同才稳定。

## 6. 快照滥用

快照适合稳定结构，不适合代替行为断言。快照变化需要能解释“为什么变化正确”。

## 7. 没有验证失败原因

RED 阶段要确认测试失败于目标缺失，而不是拼写错误、导入错误或测试环境错误。

## 8. 忽略异步和错误路径

接口、登录态、路由跳转、表单提交这类逻辑至少覆盖：

- 成功路径。
- 失败路径。
- loading/禁用状态。
- 权限或状态不满足时的行为。
