# Writing Plans

计划的目标是降低实现偏差，不是证明流程完整。

## 输入

按顺序读取：

1. 用户明确引用的需求、设计或 OpenSpec 产物。
2. 已存在的 `<FEATURE_ROOT>/<feature-id>/work.md`。
3. 当前对话中已经确认的边界。
4. 直接相关源码和项目局部规则。

先从 `.claude/rules/project-workflow.md` 获取 `feature_root` 和验证命令。不要读取 `.env*`、凭据或与任务无关的大量文件作为计划上下文。

需求仍有会改变方案、风险或验收的歧义时，只问一个阻塞问题；边界明确后再计划。

## 输出选择

- XS/S：不使用本技能，除非用户明确要求计划。
- 普通 M：默认给 2–5 个对话内结果步骤，不落盘。
- L、任意风险任务、确需跨会话恢复的 M：创建或更新 `<FEATURE_ROOT>/<feature-id>/work.md`。
- 用户明确要求独立计划文档时可以创建，但不能再为同一任务维护第二套进度事实。

`work.md` 必须使用 dev-flow 主技能定义的 State、Boundary、Context、Batches、Risk and rollback、Verification 结构。

## 批次设计

L 通常拆成 3–7 个结果导向批次。每批只写：

- Outcome：完成后可观察的结果。
- Scope：主要文件、模块或接口范围。
- Depends on：真实依赖；无依赖写 `none`。
- Verification：本批最相关命令或行为检查及预期。
- Rollback：如何独立撤回；有关联时写回撤顺序。

不要写逐行编辑教程、整段预制代码、重复背景、占位符或“适当处理”。共享契约、输入输出和错误分支必须足够明确，具体实现细节由执行时结合源码决定。

## 自审

计划完成前检查：

1. 每个验收结果至少落在一个批次。
2. 没有超出 Boundary 的批次。
3. 共享接口先于消费方，依赖和回撤顺序一致。
4. 每批都能独立验证；L 运行时行为不能只写 type-check/lint。
5. 风险标签和控制与实际改动匹配。
6. Context 只列少量非敏感恢复输入。

覆盖、计划审查或回撤检查可以作为可选自审视角；结论直接修入同一个 `work.md`，不默认生成报告。

## 门禁与交接

有风险标签时：

- `Approval` 保持 `pending`。
- 计划完成后返回 dev-flow 输出风险卡。
- 风险卡后的用户确认前不创建实现 Todo、不勾选批次、不写业务代码。

无风险时写 `Approval: not-required`，L 可直接进入第一个批次。输出只说明计划位置、批次数、主要风险和下一批；不重复完整路径清单。
