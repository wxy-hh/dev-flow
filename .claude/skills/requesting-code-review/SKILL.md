# 发起代码审查

## 输入

审查者只接收必要材料：

1. Boundary 或用户需求摘要。
2. 当前 `git diff --stat`、相关 diff 和必要调用上下文。
3. `work.md` 中与本次批次相关的风险、回撤和验证；没有工作文件时不补建。
4. 已实际运行的验证及结果。

不要把整个长对话、无关源码或预先写好的“预期问题”交给审查者。

## 审查重点

- 行为是否满足目标且没有超出范围。
- 类型、接口、状态、错误和并发/异步路径是否闭合。
- 新旧调用方、默认值和兼容分支是否一致。
- security/data/money/external/availability 风险是否被正确控制。
- 回撤边界是否与实际 diff 一致。
- 验证是否覆盖真正改变的运行时行为。
- 是否留下临时文件、调试代码、敏感日志或无引用实现。

只报告高把握且与本次改动直接相关的问题，不用风格偏好填充数量。

## 输出格式

按严重度列出：

```text
[Critical|Important|Minor] 标题
证据：文件/行或行为路径
后果：为什么会失败
修复：最小可行改动
验证：修复后如何证明
```

如果没有问题，明确说明未发现阻塞项，并指出仍未覆盖的验证，而不是笼统“代码很好”。

## 处理结果

- Critical/Important：修复后重审受影响 diff，或记录用户明确接受的风险。
- Minor：与用户目标相关时修复；否则作为非阻塞建议。
- 发现新风险标签：返回 dev-flow 更新风险卡，不能自行跨过门禁。

存在 `work.md` 时，把最终 findings、修复和残留风险压缩到 Verification 下的 `Review` 小节；不创建独立报告。审查结束后仍需完成新鲜验证。
