# dev-flow 优化意见审查稿

> 依据：真实需求使用记录（参数回溯 S、八字分享 risk-minimal-m）与过程日志。  
> 目的：把「对应问题 → 优化意见 → 是否有必要」摊开，供人工决定做 / 不做 / 延后。  
> 设计底线（审查时对照）：
>
> 1. **小任务不过度治理**
> 2. **高风险任务不能轻易越级**（含向下越级：该拦的不能省）
> 3. **工作流规则能长期维护、不持续膨胀**

**建议读法**：先看「总表」，再按条目细读；默认立场是 **能不做就不做**，只保留高杠杆、低膨胀项。

## 2. 问题与优化意见详表

### A1. CLI 裸名 / 可复制调用

| 项 | 内容 |
|----|------|
| **对应问题** | 八字分享启动时：`dev-flow-status` → exit 127；改为 `node …/dev-flow-status.mjs` 才成功。文档与仓库可执行名不一致。 |
| **证据来源** | `2026-07-21-…dev-task…txt`；`dev-flow-process-log.md` 摩擦点 1 |
| **优化意见** | 任选其一或组合：① doctor/onboard 安装无扩展名 shim；② 所有文档与 CLI 帮助统一写可复制的 `node <path>/dev-flow-status.mjs`；③ `next`/失败信息始终打印完整 argv |
| **解决什么** | 去掉路径试错税；risk-minimal 路径本身就要多次 CLI，首刀失败成本高 |
| **不解决什么** | 不减少门禁数量，不改变 policy |
| **不做的代价** | 每次新会话/新模型仍会踩一次；过程日志会持续抱怨，但**不伤正确性** |
| **膨胀风险** | 极低（脚本/安装层） |
| **审查问题** | 是否值得为「第一次就能打通」做 shim，还是只改文档即可？ |

---

### A2. security inline 证据语义过窄

| 项 | 内容 |
|----|------|
| **对应问题** | `complete-gate security_review --evidence-inline` 首次被拒：校验要求「鉴权矩阵/分支 + 验证结果」。功能是隐私白名单（非登录/鉴权），第一次合理措辞不过，第二次套「登录/鉴权/SSO 均不触及」才过。 |
| **证据来源** | process-log 摩擦点 3；`dev-flow-validate.mjs` 中 `hasSecurityInlineMatrix` |
| **优化意见** | 1）通过条件改为两类之一：Auth 矩阵类 **或** Privacy/敏感字段类（白/黑名单 + 边界 + 验证结果）；2）失败时打印推荐句式；3）`risk-gates.md` 增加短模板（控制行数，不拆新标签） |
| **解决什么** | 减少空话套矩阵；让「security = 敏感信息」与「security = 鉴权」都能留下可审计证据 |
| **刻意不做什么** | **不**新增 `privacy` 标签；**不**把 light 升 full；**不**降低「禁止只写已审查」的强度 |
| **不做的代价** | 门禁仍能拦空话，但代理会学会「套鉴权模板」→ 证据质量下降、第一次总被拒一次 |
| **膨胀风险** | 低（校验函数 + 几行模板）；若写成新标签/新 gate 则高 |
| **审查问题** | 是否接受「隐私类固定字段句式」作为与鉴权矩阵并列的合法 light 证据？ |

---

### A3. dirty workspace 与「输入材料」

| 项 | 内容 |
|----|------|
| **对应问题** | 用户事先编写的 untracked 需求设计文档触发 dirty 拦截，需 `--existing-diff unrelated --reason`。机制正确，缺常见情形示例与失败提示。 |
| **证据来源** | process-log 摩擦点 2；八字分享 start 失败输出 |
| **优化意见** | 1）`start` 失败时：若仅 `docs/**` 或 `*.md` untracked，附加推荐命令与 reason 示例；2）status-cli / guide **一处**写「需求文档属输入材料」示例 |
| **解决什么** | 降低「第一次不知道怎么声明」的摩擦 |
| **刻意不做什么** | **不**自动默认 unrelated；**不**把 docs 排除在 dirty 检测外（防偷渡语义） |
| **不做的代价** | 机制仍安全；代理多试一轮或读文档；**不伤正确性** |
| **膨胀风险** | 极低（错误字符串 + 一处示例） |
| **审查问题** | 失败提示是否足够，还是必须改文档？ |

---

### A4. `next` 输出可粘贴完整命令

| 项 | 内容 |
|----|------|
| **对应问题** | risk-minimal 约 6 次 CLI；每次需正确 feature-id 与 flags，心智负担在「拼命令」而非「想清楚门禁」 |
| **证据来源** | process-log 总评；八字分享 CLI 序列 |
| **优化意见** | `dev-flow-status next` 输出唯一 blocker 时，附带一行可复制完整命令（含调用前缀与已知参数） |
| **解决什么** | 降操作税，不减步骤语义 |
| **不做的代价** | 体验差一截，正确路径仍可走完 |
| **膨胀风险** | 低 |
| **审查问题** | 是否与 A1 一起做即可覆盖大部分 CLI 摩擦？ |

---

### B1. 无风险 S 完成质量（软检查）

| 项 | 内容 |
|----|------|
| **对应问题** | 参数回溯 S：typecheck/lint/浏览器主路径通过后宣称完成，用户随后贴出 Radix `DialogTitle` a11y console error |
| **证据来源** | `2026-07-20-…图片参数回溯…txt` 后半段 |
| **优化意见** | 仅在 SKILL「完成汇报」增加极短软约束：① 做过浏览器验证则扫本次 console；② 新增 Dialog/浮层核对 Title 等项目 a11y 约定。**不**强制 code-review / status / feature-check |
| **解决什么** | 减少「完成后用户当 QA」 |
| **若做错会怎样** | 写成强制 gate → 小任务过度治理（违反底线 1） |
| **不做的代价** | 偶发 a11y/console 漏网，靠用户或后续补丁；S 路径仍保持轻 |
| **膨胀风险** | 软检查：低；强制门：高 |
| **审查问题** | 漏网是否可接受为 S 的正常误差，还是值得加 2 行软检查？ |

---

### B2. 分级不被「对话形态」绑架

| 项 | 内容 |
|----|------|
| **对应问题** | 参数回溯先做产品设计再实现，若因此升 M 会多 code-review 与更重验证，对「假数据→真数据+回填」过重 |
| **优化意见** | 不改契约；审查/培训时强调：分级看拓扑与契约，不看是否写了长设计 |
| **建议结论** | **不做代码改动** |

### B3 / X*：明确不做的项（防「优化」伤底线）

| ID | 诱惑 | 为何不做 | 对应底线 |
|----|------|----------|----------|
| B3 | 合并轻量 M 与 risk-minimal-m 以少 CLI | 无风险轻量 M 无实现前风险卡/审批；合并等于高风险向下越级 | 2 |
| X1 | 所有 S 强制 code-review | 用全量审查修偶发 a11y，成本不对 | 1 |
| X2 | 无风险 S 建 status.md | 资产与 feature-check 蔓延 | 1、3 |
| X3 | 新增 `privacy` 标签 | 与 security 重叠；标签表与并集门禁变复杂 | 3 |
| X4 | 有需求文档 → 标准 M | 八字分享证明 light+risk 已够；文档厚度≠规模 | 1、2 |
| X5 | security light 一律 full 报告 | 抬高 risk-minimal，鼓励绕过 | 1、2 |
| X6 | plan-review 并入 code-review | 破坏实现前/后分工 | 既有底线 |

| ID | 对应问题 | 优化意见 | 建议 |
|----|----------|----------|------|
| C1 | 同一事实写进 SKILL / guide / status-cli / protocol 多份 | 契约只在 contract+validator；人话细则单文件；guide 只链 | 约定，有改文档时遵守 |

## 3. 实录对照：哪些「痛」不该优化掉

| 体验 | 是否保留 | 理由 |
|------|----------|------|
| dirty workspace 拦截 | **保留** | 防实现前业务 diff 偷渡；只补提示（A3） |
| 实现前 HUMAN GATE | **保留** | 隐私/设计决策需人确认；过程日志评价顺畅 |
| security 要具体证据 | **保留强度** | 逼出白名单是八字分享最有价值决策；只改语义（A2） |
| 行为验证要运行时证据 | **保留** | Portal 时序、导出 404 单测盖不住 |
| risk-minimal 多步 CLI | **保留步骤** | 可减操作税（A1/A4），不合并 gate |
| 无风险 S 无 status | **保留** | 参数回溯路径证明轻量价值 |

## 4. 建议决策包（供勾选）

### 4.1 若只做「有必要」的最小包

- [ ] **A1** CLI 可调用（shim 或统一完整命令）
- [ ] **A2** security inline 隐私类证据 + 失败模板
- [ ] **A3** dirty 失败推荐 reason（docs-only）
- [ ] **A4** `next` 可粘贴命令（可与 A1 合并交付）

预期：门禁语义不变；八字类任务少 1 次 evidence 重试、少路径试错；文档/契约几乎不涨。

### 4.2 可选加项

- [ ] **B1** S 完成软检查（仅 SKILL 数行，禁止升级为 gate）

### 4.3 明确排除

- [x] 不做 X1–X6、B3  
- [x] 不把过程摩擦解释成「要加更多 Markdown 资产」

### 4.4 不单独立项

- [ ] C1/C2：有文档或 A2/A3 改动时顺带遵守  
- [ ] B2/C3：纪律与范围，无代码任务

## 7. 一句话立场（供拍板）

> **优化空间主要在「同样的门更好过」和「失败更可操作」，不在「再加一层治理」。**  
> 若审查后认为 A1–A3 的「不做代价」可接受，可以全部不做——**系统仍然正确**；做则改善真实使用摩擦，且不牺牲三条设计底线。
