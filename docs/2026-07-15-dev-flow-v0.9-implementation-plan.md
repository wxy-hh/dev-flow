## 一、升级目标与硬约束

本次只发布一个版本，不拆分 v0.9.0/v0.9.1：

```text
workflow_version: 0.9.0
status_schema: 3
authorization_schema: 1
completion_schema: 1
```

`status_schema` 与 `completion_schema` 保持不变。本次不增加 status 生命周期字段，不增加 seal，不把 cleanup inventory 写入 completion。只有 write-authorization 首次增加明确 schema，因为它将承载审批绑定信息。

### 1. 必须解决

1. standard L 携带风险标签时不能在 init 阶段被尚未生成的 full report 卡死。
2. implementation approval 必须绑定当时的任务分类、风险、计划资产和受保护写入范围。
3. Edit、Write 与 Bash 不能读取不同的授权事实，Bash 不得静默绕过待审批状态。
4. 静态审查、代码阅读、lint/type-check 不能冒充实际运行验证。
5. 高后果标签提高证据强度，但不能把 risk-minimal 隐式升级成 standard L。
6. feature-check 通过并生成有效 completion 后即可进入 Git；compact/full 只做资产维护。
7. compact 删除 untracked 文件前必须取得独立、精确、绑定 inventory 的第二次确认。
8. 用户明确确认的外部需求可以直接登记为 HUMAN GATE evidence 并免于重复询问；文档标题或代理推断不能代替用户确认。
9. v0.9 upgrade 遇到 v0.8 活动 status 必须零修改拒绝。
10. 规则修订应优先修改行为、错误信息和单一规范源，不重做资产模型。
11. 标准 M/L 的三条需求固化路线必须在分类同回合写入 status，不能只存在于对话 HANDOFF 中。
12. required HUMAN GATE 不保留可被误解为授权的 `skipped` 状态；接受残留风险仍按确认处理并保留原话 evidence。
13. 轻量 L 的行为验证必须由机器策略强制为 full，不能只在 smoke/skill 文案中声明。
14. risk-minimal 与轻量 L 必须先完成已派生的 rollback/security 实现前义务，再展示 implementation approval。
15. finish guard 绑定刚完成 feature 后仍须扫描其他 active status，不能被 closed authorization 提前短路。
16. 标准 M 的 requirements-coverage 仅按风险触发；只有标准 L 使用固定 coverage 骨架。

### 2. 必须保持

- 无风险 XS/S 不创建 status、需求、计划或审查资产。
- risk-minimal XS/S 不触发 requirements coverage 或 plan-review。
- 现有 requirement、plan、coverage、rollback、review、verification 资产结构继续有效。
- 决策与计划修订写入现有计划正文，不增加 `pending_decisions`、`revisions` 等 status 字段。
- risk label 是语义和审计信息；即使没有增加 gate 强度也不得自动删除。
- code-review 仍不是 risk gate，不允许 `promote-gate code-review`。
- business fingerprint 的现有排除范围不变。
- compact 仍表示删除中间资产，full 仍表示归档中间资产。
- 历史 completion 继续只读兼容。
- 实现计划唯一正式名称为 `<FEATURE_ROOT>/<feature-id>/实现计划.md`；本仓库不保留旧计划文件名兼容分支。

### 3. 明确不做

- 不强制统一为 `work.md`，不增加固定七章 heading 契约。
- 不新增任务等级、风险标签或通用 gate。
- 不新增 seal/reopen 状态机。
- 不新增 completion schema 2 或 cleanup inventory 字段。
- 不维护大规模 Bash 命令白名单或通用 shell 副作用解析器。
- 不用自然语言扫描自动判断 finding severity。
- 不把 coverage 变成实施后证据表。
- 不因 untracked 文件存在而强制 full 或要求先 commit。
- 不实现 v0.8 活动任务到 v0.9 的在途迁移。

## 二、v0.9 目标行为

### 1. 阶段化 gate

| 阶段 | 允许状态 | 阻塞条件 |
|---|---|---|
| `current/init` | required gate 可为 pending | schema、route、标签和 evidence 形态非法 |
| `approval` | 实现前义务必须完成 | requirement、plan、coverage、plan-review、rollback/security 前置义务缺失 |
| `finish` | 完成义务必须闭环 | code-review、behavior、manual/automated verification、accepted risk 不完整 |

full gate 在 init 阶段可以声明为 pending，不要求 report 已存在；进入对应实际门禁时再要求 report 或唯一 gate evidence。

### 2. 审批与写入授权

```text
分类与计划
  → 实现前 gate 完成
  → confirm-human implementation_approval --status confirmed
  → 计算 approval basis
  → write-authorization: approved
  → Edit/Write/Bash 使用同一 authorization validator
```

required HUMAN GATE 只允许 `confirmed`：

- `requirement_confirmation` 与 `implementation_approval` 都拒绝 `skipped`。
- 用户明确表示“接受风险并继续”属于一次确认，应记录为 `confirmed` 并保留原话 evidence。
- manual-test 步骤的 `skipped` 仍只属于 partial 验证契约，与 HUMAN GATE 状态无关。

### 3. 需求固化、同回合 status 与可恢复路由

标准 M/L 在分类完成的同一回合执行 `dev-flow-status init`，并显式写入需求入口：

| 需求现状 | `--entry-gate` | 后续路线 |
|---|---|---|
| 需求模糊或无文档 | `req-probe` | `req-probe → grillme → requirement_confirmation` |
| 项目/用户明确使用 OpenSpec | `openspec` | `openspec → grillme → requirement_confirmation` |
| 已有完整但未确认文档 | `grillme` | `grillme → requirement_confirmation` |
| 用户已明确确认且无未决问题 | `writing-plans` | 登记 requirement confirmation evidence → `writing-plans` |

状态推进规则：

1. `req-probe`、OpenSpec、`grillme`、`writing-plans` 等 owner skill 完成稳定阶段后必须调用现有 `complete-gate`。
2. `current_gate` 表示最近进入或完成的稳定 gate；`next_action` 由 CLI 结合 route obligations 派生，不允许 skill 手写猜测。
3. `req-probe`/OpenSpec 的下一步固定为 `grillme`；`grillme` 的下一步固定为 `await requirement_confirmation`。
4. 标准 M 完成 writing-plans 后，仅在 coverage 被风险维度触发时进入 requirements-coverage，否则直接进入 plan-review。
5. 标准 L 固定执行 `writing-plans → requirements-coverage → plan-review`。
6. 三条标准 M/L 路线都只产生一次 requirement confirmation，且 plan-review、implementation approval 不因外部需求已确认而省略。
7. 正式计划统一保存为 `<FEATURE_ROOT>/<feature-id>/实现计划.md`；不探测、不迁移、不生成旧计划文件名。

risk-minimal 与轻量 L 同样在分类同回合 init/activate，但不传 `--entry-gate`：

```text
风险/边界卡
  → init/activate（approval-pending）
  → rollback/security 等实现前 gate（按派生结果）
  → [HUMAN GATE:implementation_approval]
  → confirm-human --status confirmed
  → 实现
  → code-review + behavior verification
```

轻量 L 的 route minimum 固定为 `rollback_units: light`、`behavior_verification: full`；风险标签只按 max 规则继续提高，不得把 full 行为验证降为 light。

### 4. 逻辑完成与资产收尾

复用现有 check-ok，不增加 seal：

```text
complete-verification
  → feature-check --finish（生成 check-ok）
  → 生成/更新 feature.md 与 completion.md
  → final-assets validator
  → logic-complete
  → Git 允许
  → compact/full 可选
```

logic-complete 必须同时满足：

1. status `--stage finish` 通过。
2. check-ok 的 workflow version、outcome、fingerprint 与当前状态一致。
3. feature.md/completion.md 通过当前版本 final-assets validator。
4. completion outcome 为 verified，或 partial 且 accepted-risk 三方链路完整。

逻辑完成不删除 status，不关闭同 feature 的补验证路径：

- 用户选择 `not now` 时不写任何新状态，也不阻塞 Git。
- 业务 diff 在 logic-complete 后变化时，check-ok 自动过期。
- 同一 feature 可以修正实现，重新运行 verification、feature-check 并更新 completion。
- 不要求创建新 feature ID。
- 新任务 init/authorize 时仍由现有逻辑关闭上一份 runtime authorization。
- finish guard 先校验 authorization 指向的刚完成 feature，再继续扫描其他带 `status.md` 的 active feature；任一未 logic-complete 都必须阻塞 Git。

### 5. compact 的 untracked 安全边界

dry-run 必须将每个资产分类：

```text
[keep] <path>
[delete][tracked] <path>
[delete][untracked] <path>
[archive] <source> -> <destination>
[conflict] <path>
```

compact 存在 untracked 删除候选时：

1. 普通 `--confirm --inventory <sha>` 必须拒绝且零修改。
2. dry-run 输出：

   ```text
   Untracked delete token:
   DELETE-UNTRACKED:<inventory-sha>:<count>
   ```

3. 当前回合停止，等待用户原样回复完整 token。
4. 执行命令必须同时包含：

   ```text
   --confirm
   --inventory <inventory-sha>
   --confirm-untracked "DELETE-UNTRACKED:<inventory-sha>:<count>"
   ```

5. token、数量、inventory 任一不匹配都零修改退出。
6. 用户也可改选 `retain full` 或 `not now`。

这项保护不改变 compact/full 的语义，也不要求先提交 untracked 文件。

## 三、数据与接口契约

### 1. contract.json

```json
{
  "workflow_version": "0.9.0",
  "status_schema": "3",
  "authorization_schema": "1",
  "completion_schema": "1",
  "human_gate_statuses": ["pending", "confirmed"]
}
```

高后果最低 risk gate：

```json
{
  "critical_correctness": {
    "behavior_verification": "full"
  },
  "irreversible_consequence": {
    "rollback_units": "full",
    "behavior_verification": "full"
  }
}
```

route policy 额外派生：

| 情况 | 实现前/完成义务 |
|---|---|
| lightweight L | rollback light、behavior full、code-review light |
| 任一风险标签 | code-review 至少 light，其他 gate 与 route minimum 取 max |
| `critical_correctness` | behavior full、code-review full |
| `irreversible_consequence` | rollback full、behavior full、code-review full |

高后果标签不直接提升 `plan_review`。plan-review 继续由任务规模、profile、模块跨度和既有规则决定：

- risk-minimal 不触发 plan-review。
- standard M/L 取路线默认强度。
- 其他标签或实际 finding 可以通过现有 promote 机制提升。

### 2. authorization schema 1

`.claude/runtime/dev-flow/write-authorization.json`：

```json
{
  "schema_version": "1",
  "workflow_version": "0.9.0",
  "feature_id": "feature-id",
  "level": "L",
  "profile": "standard",
  "state": "approved",
  "protected_roots": ["src"],
  "approval_basis": {
    "algorithm": "sha256",
    "hash": "<canonical-hash>",
    "route_hash": "<route-hash>",
    "assets": [
      {
        "path": "<repo-relative-path>",
        "kind": "plan",
        "sha256": "<content-hash>"
      }
    ]
  },
  "created_at": "<ISO-8601>",
  "approved_at": "<ISO-8601>",
  "closed_at": null
}
```

route hash 输入：

- feature ID、level、profile。
- topology、classification evidence result。
- 排序后的 risk labels。
- 派生 risk gates 与 code-review obligation。
- protected write roots。
- requirement confirmation 状态和 evidence。
- approval 时已注册的前置资产路径集合。

asset hash 输入：

- `kind: requirement|plan|spec` 的注册资产。
- requirements-coverage、plan-review、rollback-units、security-review 引用的 gate evidence。
- 路径使用仓库相对 POSIX 形式并排序。
- 文件使用原始字节 SHA-256。

排除：

- validation 命令、fingerprint、时间字段。
- code-review 和 verification 等实施后证据。
- current gate、next action 等执行进度。
- implementation approval 自身 evidence。

变更分类与恢复：

| mismatch | 用户可见结果 | 恢复动作 |
|---|---|---|
| route、risk、protected roots 或资产路径集合变化 | `APPROVAL_BASIS_STRUCTURAL_CHANGE` | 补齐新路线要求的前置 gate，再重新 confirm-human |
| 仅已批准资产内容 hash 变化 | `APPROVAL_BASIS_CONTENT_CHANGE` | 用户查看变更后重新 confirm-human；不自动重跑全部前置 gate |
| 仅实施后证据或 validation 变化 | authorization 保持有效 | 无 |

不得静默刷新 hash，不增加 `rebind-approval` 命令。所有重新绑定仍复用 implementation approval HUMAN GATE。

### 3. manual-test 兼容契约

保留现有七列表格：

```text
ID | 操作 | 预期 | 结果 | 实测 | 证据 | 风险 ID
```

frontmatter 增加可选 method：

```yaml
manual_test_steps:
  - id: MT-001
    method: browser
    result: passed
    risk_id: null
    observed: "实际观察结果"
    evidence: "截图、trace、日志、命令或测试标识"
```

允许：

```text
browser
device
api
cli
automated
```

规则：

- 所有 passed 行都必须具有非空 observed 和 evidence。
- behavior verification 为 full 时，每个 passed step 必须在 frontmatter 中具有 method。
- light 时缺 method 产生 WARN，不直接破坏旧七列表。
- `automated` 必须在 evidence 中给出实际执行命令与测试标识，并有已注册 verification 资产。
- `static-review`、代码阅读、lint、type-check 不能作为 passed method。
- failed 必须记录 method 和 observed。
- skipped 必须关联唯一 AR，最终 outcome 为 partial。
- 尚未形成手测步骤、status accepted_risks、partial-acceptance 三方一致时，不得完成 verification gate；闭环后允许把 `verification-before-completion` 写入 completed_gates，并以 `outcome: partial` 正常进入 logic-complete/Git。
- frontmatter 与七列表同时存在时，ID、result、risk_id 必须一致。
- manual-test 中只记录实际执行或实际自动化验证的步骤；纯 code-review finding 留在 review 资产。

### 4. validator/CLI 变化

新增只读接口：

```text
dev-flow-validate approval-basis <root> <status-path>
dev-flow-validate authorization <root> <authorization-path>
```

finalizer 新增：

```text
--confirm-untracked <exact-token>
```

status CLI 收紧/扩展现有命令，不新增状态机命令：

```text
dev-flow-status init ... [--entry-gate req-probe|openspec|grillme|writing-plans]
dev-flow-status confirm-human <feature-id> <gate> --status confirmed --evidence <user-evidence>

```

- 标准 M/L（非轻量 L）缺少或传入非法 `--entry-gate` 时 init 零修改失败。
- risk-minimal/轻量 L 禁止传 `--entry-gate`，初始 gate 由派生出的实现前义务决定。
- `complete-gate` 保留已完成 gate，并从 policy obligations 计算 `next_action`。
- `confirm-human` 拒绝 `skipped`；接受残留风险必须写为 `confirmed` + 原话 evidence。

不新增：

```text

dev-flow-status reopen
dev-flow-status rebind-approval
feature --stage sealed
completion schema 2
```

## 四、精确实施任务

每个任务均按以下 TDD 顺序执行：

1. 添加目标行为测试。
2. 单独运行并确认测试因目标缺陷失败。
3. 实现最小行为修改。
4. 运行目标测试。
5. 运行所有受影响测试。
6. 全绿后提交，不提交已知失败测试。

### Task 1：阶段化校验与风险路线

主要文件：

- `.claude/skills/dev-flow/contract.json`
- `.claude/skills/dev-flow/scripts/dev-flow-policy.mjs`
- `.claude/skills/dev-flow/scripts/dev-flow-validate.mjs`
- `.claude/skills/dev-flow/scripts/tests/dev-flow-status-test`

实施：

1. 将 workflow version 改为 0.9.0。
2. 保持 status/completion schema 版本。
3. 增加 authorization schema 1。
4. current/init 阶段允许 full gate pending，不要求 report。
5. approval 阶段只强制实现前 process gates。
6. finish 阶段强制 code-review、behavior、verification 和 accepted risk。
7. contract 的 HUMAN GATE 状态收敛为 pending/confirmed；两个 required gate 都拒绝 skipped。
8. risk-minimal/standard 路线按前述高后果矩阵派生 code-review 和 risk gates。
9. 标签没有提高最终 gate 强度时输出 INFO，不改变标签。
10. contract 与 risk-gates 文档映射不一致时 doctor/test 失败。
11. 轻量 L route minimum 把 behavior verification 固定为 full，并保留 rollback light、code-review light。
12. 标准 M/L init 必须接收合法 `--entry-gate`；risk-minimal/轻量 L 的初始 gate 取首个已触发实现前义务。

测试矩阵：

```text
standard L × 7 risk labels
standard M × 7 risk labels
risk-minimal XS × 7 risk labels
risk-minimal S × 7 risk labels
lightweight L × 7 risk labels
```

使用 policy 表驱动测试覆盖 5 条 status-bearing route × 7 个标签，共 35 个组合；集中断言 route minimum、标签 minimum、code-review 强度和 standard M 不被强制 coverage。端到端 fixture 不复制 35 份，而是每条路线保留代表性用例，覆盖：

```text
init succeeds
approval blocks before upstream evidence
approval succeeds after evidence
finish blocks before completion evidence
finish succeeds after evidence
```

特别断言：

- critical/irreversible risk-minimal 不出现 plan-review。
- critical 要求 behavior full + code-review full。
- irreversible 要求 rollback full + behavior full + code-review full。
- required HUMAN GATE skipped 被拒绝，implementation approval 不产生 approved authorization。

提交：

```text
fix(dev-flow): make v0.9 gates stage-aware and route-sensitive
```

### Task 2：审批绑定与统一写入授权

主要文件：

- `.claude/skills/dev-flow/scripts/dev-flow-validate.mjs`
- `.claude/skills/dev-flow/scripts/dev-flow-status.mjs`
- `.claude/hooks/dev-flow-gate-guard.sh`
- `.claude/skills/dev-flow/scripts/tests/dev-flow-hooks-test`

实施 approval basis：

1. validator 实现唯一 canonicalization 函数。
2. `approval-basis` 输出 hash、route_hash、asset manifests。
3. confirm-human/activate 复用该接口写 authorization。
4. `authorization` 重算并比较 hash。
5. 比较结果区分 structural/content mismatch。
6. stale authorization 不能因为 JSON 中仍写 `state: approved` 而放行。
7. 重新确认复用现有 HUMAN GATE，不增加新命令。

实施 Bash 守卫：

1. `.claude/settings.json` 的 Bash matcher 先运行 gate guard，再运行 finish guard。
2. Edit/Write 继续按 protected roots 判断。
3. Bash 不解析目标文件，也不维护常用命令白名单。
4. authorization 为有效 classified/approved 时，普通 Bash 保持现有行为。
5. pending、missing、closed、unknown、stale 时：
   - 单一、无 shell 控制符的 dev-flow status/validate/policy/doctor 控制命令自动放行。
   - Git add/commit/push 交给 finish guard，不在 gate guard 决定。
   - 其他 Bash 在 strict 模式 deny，在 ask 模式 ask，在 off 模式维持关闭。
6. 含重定向、命令替换、链式执行的命令不能伪装成 dev-flow 控制命令。
7. 无风险 XS/S 一旦 authorize 为 classified，后续不增加命令白名单摩擦。

测试：

| 场景 | 预期 |
|---|---|
| approved + basis 匹配 | Edit/Write/Bash 允许 |
| approved + plan 内容变化 | 三类工具均阻塞并报 content change |
| approved + risk/roots 变化 | 三类工具均阻塞并报 structural change |
| pending + shell 重定向 | strict deny / ask 模式 ask |
| pending + dev-flow status 单命令 | 允许 |
| pending + dev-flow 命令拼接 shell 写入 | 阻塞 |
| classified XS/S | 保持轻量 |
| Git finish 命令 | 由 finish guard 决定 |

### Task 8：全流程复审收口（3 个 P1、6 个 P2）

本任务记录 2026-07-16 在初版 v0.9 实施完成后进行的端到端复审。修正直接并入 v0.9.0，不另拆 v0.9.1，也不引入新 schema、新命令或新文档层级。

#### P1-1：required HUMAN GATE 仍存在 skipped 旁路

问题：`executing-plans` 和协议示例仍允许把 required implementation approval 记为 `skipped`，与 validator 的强制确认语义冲突。

修正：

- `contract.json` 的 HUMAN GATE 状态只保留 `pending`、`confirmed`。
- `dev-flow-status confirm-human` 只接受 `--status confirmed`。
- requirement confirmation 与 implementation approval 都只有 `confirmed` 才满足义务。
- 用户拒绝、延期或选择跳过时保持 `pending`，记录在对话或计划正文，不伪造 gate 完成。
- 同一条回复不得同时充当 requirement confirmation 和 implementation approval。

验收：两类 required HUMAN GATE 的 `skipped` 调用都必须失败，且不得生成 approved authorization。

#### P1-2：finish guard 只检查授权 feature

问题：存在已完成且已授权的 feature 时，guard 检查该 feature 后提前退出，可能漏掉同仓库其他仍 active、未完成的 status。

修正：优先校验授权 feature，但随后继续扫描所有其他 active status；只有全部活动任务均满足完成条件才允许 Git 写操作。

验收：`closed alpha + incomplete beta` 必须阻塞，错误信息指向 beta；不得因 alpha 已通过而提前放行。

#### P1-3：lightweight L 的行为验证强度不足

问题：轻量 L 仍被派生为 `light` behavior verification，低于 L 级任务最低验证强度。

修正：policy 将 lightweight L 的 behavior verification 固定为 `full`，文档、fixture、validator 和测试同步。

验收：lightweight L 缺 full behavior evidence 时 approval/finish 必须失败。

#### P2-1：risk-minimal 与 lightweight L 未在同回合建立 status

问题：先展示 HUMAN GATE、后建立 status 会丢失当前 gate、风险义务和恢复位置；rollback/security 等前置风险 gate 也可能被 implementation approval 越过。

修正：路由确定后、展示首个 gate 前必须在同一回复执行 `init`。risk-minimal 和 lightweight L 的 `current_gate` 指向首个未满足的前置风险 gate；没有前置风险 gate 时才指向 implementation approval。

验收：irreversible、security 等路由 init 后首先显示对应风险 gate，完成后才进入 implementation approval。

#### P2-2：standard M/L 的 requirements coverage 规则混用

问题：标准 M 被无条件要求 coverage，造成中任务过度治理；标准 L 的固定 coverage 又可能在条件描述中被弱化。

修正：standard M 仅在分类或风险审查明确触发/提升 requirements coverage 时启用；不能因为任务属于 standard M 就自动增加 coverage。standard L 固定要求 coverage。writing-plans 完成后的 `next_action` 必须据此派生。

验收：无 coverage 义务的 standard M 从 writing-plans 进入 plan-review；standard L 必须先进入 requirements coverage。

#### P2-3：partial outcome 无法区分“合规完成”与“未完成”

问题：所有 partial 都被视为未完成，会让已经明确缩减范围且满足三方一致的任务无法收口。

修正：仅当“交付范围、计划/需求、验证证据”三方一致时，partial 可完成 verification；任一方仍声称原范围完成时保持未完成。

验收：compliant partial 可完成 verification，incomplete partial 必须阻塞。

#### P2-4：requirement confirmation 仍可被自动 skipped

问题：部分入口技能将已有文档或外部确认等价为 skipped，导致 required gate 的语义不稳定。

修正：外部确认统一通过 `confirm-human --gate requirement_confirmation --status confirmed` 固化；已有需求文档只决定从 grillme 开始，不代表已经确认；只有用户明确确认且无未决问题时才可从 writing-plans 入口开始。

验收：自动 skipped 不得通过；外部明确确认必须保留 evidence 并进入 writing-plans。

#### P2-5：需求入口与 current_gate/next_action 不可恢复

问题：standard M/L 的三种需求路径未写入状态，compact 或换代理后只能依靠聊天上下文猜测恢复点。

修正：`init --entry-gate` 支持并校验 `req-probe`、`openspec`、`grillme`、`writing-plans`；`complete-gate` 按 policy 更新 `current_gate` 与 `next_action`。三种标准路径统一为：

```text
已有完整需求文档：grillme → 更新需求文档 → requirement_confirmation → writing-plans
需求模糊或没有文档：req-probe/openspec → grillme → requirement_confirmation → writing-plans
用户明确确认且无未决问题：writing-plans
```

验收：每个入口 init 后的 current gate 正确；完成 req-probe/openspec、grillme、writing-plans 后均能从 status 唯一恢复下一动作。

#### P2-6：route × risk 测试不足

问题：只测试少量代表路径，policy 变更可能让某个标签在某条路线静默降级。

修正：增加 5 条持久化路线 × 7 个风险标签的 35 组合表驱动测试，并保留三种入口、关键 gate 转换与代表性端到端测试。

验收：35 个组合全部验证 approval、behavior、rollback、code-review、coverage 和初始 gate 派生；任何风险义务降级都必须使测试失败。

| exact untracked token | inventory 未变化时可执行 |
| token/inventory 漂移 | 零修改失败 |
| 改选 full | untracked 被归档而非删除 |
| not now | 不修改、不阻塞 Git |

### 5. 逻辑完成可恢复

| 场景 | 预期 |
|---|---|
| check-ok + completion 有效 | 未 finalization 也可 Git |
| fingerprint 漂移 | Git 阻塞 |
| 同 feature 重新验证 | 恢复 logic-complete |
| completion outcome 错配 | Git 阻塞 |
| 新任务已激活 | 旧 feature stamp 不授权新任务 |
| 三方一致的 partial | 可完成 verification 并继续收口 |
| 范围、计划、证据不一致的 partial | 保持未完成并阻塞收口 |

### 6. 规则长期维护

- contract/policy 表驱动测试覆盖 5 条持久化路线 × 7 个风险标签，共 35 个组合。
- doctor 能发现风险映射和版本漂移。
- protocol 之外不复制完成状态机。
- status CLI 独占状态写入和 next_action 转换；policy 独占 route 派生。
- 不新增 work/seal/reopen/finalization status。
- 新增规范不得通过提高 line budget 解决重复。
- 实施完成后，核心规范文档新增行应由删除重复规则抵消；测试代码不计入。

## 八、v0.9 最终变更清单

本节以最终仓库状态为准，用于执行者逐项核对，不要求按文件重复理解规则。机器行为仍由 contract、policy、status CLI、validator 和 hooks 单源拥有。

### 1. 路由与治理

- 保留无风险 XS/S 零流程成本，不建立 status。
- risk-minimal XS/S 和 lightweight L 在首个 gate 前同回合建立 status，但不升级为 standard M/L 全套流程。
- standard M/L 支持 `req-probe`、`openspec`、`grillme`、`writing-plans` 四个可校验入口。
- 统一三种需求路径：模糊需求先 probe，完整文档先 grillme，明确确认且无未决问题直接 writing-plans。
- standard M 的 requirements coverage 条件启用；standard L 固定启用。
- lightweight L 的 behavior verification 固定为 full。
- rollback/security 等前置风险 gate 必须在 implementation approval 前完成。
- route × risk 义务由 policy 派生，status 和文档不得复制另一套映射。

### 2. HUMAN GATE 与写入授权

- required HUMAN GATE 只接受 `confirmed`；`skipped` 不再是合法完成状态。
- requirement confirmation 与 implementation approval 分阶段确认，同一条回复不得复用。
- 用户明确确认且无未决问题时可跳过澄清技能，但仍由已确认事实建立 writing-plans 入口。
- 外部确认通过 status CLI 写入 confirmed 和 evidence，不由技能自动推断 skipped。
- 只有全部 required HUMAN GATE confirmed 后才能生成 approved authorization。
- plan、risk、project root 或 gate 变化后 authorization 按既有 stale 规则失效。

### 3. 状态恢复与资产

- `init --entry-gate` 写入真实首个 gate；非法或缺失入口直接失败。
- `complete-gate` 由 policy 派生并写回 `current_gate`、`next_action`，compact 后无需猜测聊天上下文。
- `req-probe/openspec → grillme → requirement_confirmation → writing-plans` 的转换可由 status 唯一恢复。
- writing-plans 产出的正式计划资产统一命名为 `实现计划.md`。
- 不保留旧计划文件名、检测分支或迁移兼容逻辑；其他项目同步 v0.9 后直接使用新名称。

### 4. 验证、审查与完成

- full manual-test 必须包含 method；七列 light manual-test 继续兼容。
- automated verification 只有存在真实执行命令及成功证据时才可通过。
- static review 不得冒充 passed code review。
- compliant partial 在交付范围、计划/需求、验证证据三方一致时可完成；不一致时保持未完成。
- logic-complete 与 finalization 解耦，选择 not now 不阻塞 Git。
- finish guard 先检查授权 feature，再扫描所有其他 active status，防止多任务仓库漏检。

### 5. finalization 与 upgrade

- compact 删除 tracked 资产沿用普通 inventory confirmation。
- compact 涉及 untracked 时要求与当前 inventory 绑定的 exact token；漂移时零修改失败。
- full 模式归档而非删除 untracked 资产。
- v0.8 存在 active status 时 upgrade 在修改任何托管文件前拒绝；无活动任务时幂等升级。
- 历史 completion/archive 保留，旧 runtime authorization 不迁移成 approved。
