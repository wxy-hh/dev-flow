# 对 dev-flow v0.9 实施计划的评审意见

**评审对象**：`docs/2026-07-15-dev-flow-v0.9-implementation-plan.md`  
**用途**：可直接转发给计划作者（或 GPT）改写 v0.9 实施范围  
**日期**：2026-07-15

**对照材料**：

- `2026-07-15-cross-modal-relay-dev-flow-experience.md`（真实标准 L 体验）
- `docs/2026-07-15-dev-flow-optimization-from-cross-modal-experience.md`（优化清单 O-01～O-18）
- 设计底线三条：
  1. 小任务不过度治理
  2. 高风险不能轻易越级
  3. 规则可长期维护、不持续膨胀

**总评一句话**：方向有若干正确内核，但整体是平台重写级 0.9，不是「按体验摩擦修关节」。应保留 4–5 个手术点，砍掉资产模型重做与过激加码，分批交付。

---

## 一、同意保留的部分

### 1. stage-aware gate 校验（Task 1）

- **原因**：体验中 standard L 打 `data` 标签时，init 阶段就被 `risk_evidence must use report mode when gate is full` 拦住；报告尚未生成，属于晚绑定失败。
- **意见**：必须做。init 允许 full gate pending；approval/finish 再分别收紧。不要用「自动删标签」解决冲突。

### 2. 逻辑完成与 compact/full 解耦（Task 7/8 的目标，不是其重型实现）

- **原因**：体验中 complete-verification → feature-check → finalize 互相咬合；not now / fingerprint 漂移导致 Git 与收尾阻塞，循环重跑。
- **意见**：目标必须保留——任务逻辑完成应允许进入 Git；compact/full 只是可选资产维护。实现方式见下文「应改轻」。

### 3. dry-run 标注资产去向（Task 8 输出格式）

- **原因**：体验阶段 8：dry-run 只列路径，不区分 delete/archive/keep，执行者无法预判 compact 会删什么。
- **意见**：必须做。输出应含 `[keep]` / `[delete]` / `[archive]` 等。

### 4. approval basis 的「思路」（Task 3 目标）

- **原因**：高风险不能轻易越级——implementation_approval 后若可随意改计划/风险范围/受保护根，审批就脱钩。
- **意见**：方向正确，但粒度过猛（见反对项）。应做简化版 basis，而不是全文字节级过敏哈希。

### 5. 外部确认必须走 confirm-human，禁止标题 auto-skip（Task 9）

- **原因**：体验阶段 0/0.5：执行者仅凭「状态：需求已确认」跳过 grillme，却没有把用户确认登记为 requirement_confirmation evidence，用户本人都不确定是否合规。
- **意见**：同意。用户明确确认且无未决问题时可以跳过 req-probe/grillme，但必须通过 confirm-human 登记 HUMAN GATE evidence；plan-review / implementation_approval 不因「需求已确认」省略。

### 6. 不因标签无增量而自动剥标签

- **原因**：标签是语义与审计，不只是 gate 增量；体验中标准 L 已覆盖 data 的 light 要求，但自动剥标会削弱可追溯性。
- **意见**：同意。无增量时 INFO 即可，禁止自动删除标签。

### 7. 升级时拒绝静默接管 v0.8 活动任务（Task 11）

- **原因**：活动 status 被静默改写会导致在途任务证据断裂。
- **意见**：同意。发现活动 status 则零修改拒绝升级。

### 8. 不新增 status 字段 / 不新增通用 gate 名

- **原因**：符合「规则不膨胀」；体验建议的 `pending_decisions` / `revisions` 字段应拒绝。
- **意见**：同意。决策与修订写计划正文即可。

## 二、不同意或必须改写的部分

### 1. compact 对 untracked 只「告知删除」、不硬保护 —— 必须改

- **计划现状**：§1.1.7 / §1.3「不要求 untracked 强制 full」；Task 8 输出 `[delete][untracked]`，语义仍是 compact 可永久删除未跟踪文件。
- **原因**：
  1. 体验阶段 8 的核心事故不是文案不清，而是 untracked 的 manual-test / verification / code-review 被删后不可从 git 恢复。
  2. 只「明确告知会永久删除」不等于安全边界；代理 confirm 后仍会丢数据。
  3. 这是体验全流程唯一接近真实数据丢失的 bug 级问题。
- **意见**：
  - dry-run 必须标注 `[delete][untracked]`（计划这部分可留）。
  - **confirm 默认拒绝删除 untracked**，或强制改为 archive，或要求第二次精确确认词（如整行 `delete untracked`）。
  - 「不强制改用 full」可以保留，但禁止在无二次确认时对 `??` 文件执行删除。
  - 若不改此项，等于体验 P0（O-01）未修，只修了文案（O-02/O-03）。

---

### 2. 强制收敛为单一 `work.md`（Task 2）—— 建议砍掉

- **计划现状**：需求 / coverage / 计划 / 回撤 / 决策统一进 `work.md`；用固定 heading 顶替独立 coverage/rollback 资产；risk-minimal 也要精简 work。
- **原因**：
  1. 体验摩擦是格式晚绑定、跳过依据不清、收尾危险，**不是**「计划期文件太多」。
  2. coverage 并进 work 后，矩阵与计划同一作者同一文件，削弱「覆盖可被独立核对」的价值；体验阶段 3 的 REQ-013 虚标，正依赖可对照的覆盖物。
  3. 固定多段英文 heading + validator 校验 = 新仪式，抬高 risk-minimal / 小任务成本，违反「小任务不过度治理」。
  4. 用现有 `kind: plan` 冒充 work，语义含糊，后续 doctor/统计更难。
- **意见**：
  - **不要**做资产模型重做，不要强制 `work.md` 七章契约。
  - 多文件结构可保留；轻量路径本来就可以少产物。
  - 决策与 D 修订：在计划正文加一小节即可（对应 O-17），不进 status schema。
  - coverage full 仍应可独立审计（独立文件或至少可被 review 引用的独立结论），不要为「收敛」牺牲可审计性。

---

### 3. approval basis 全文内容哈希过猛（Task 3）—— 保留目标，缩小粒度

- **计划现状**：work 全文、requirement/spec、plan-review disposition、rollback heading、风险标签、protected roots 等全部进入 canonical hash；任一内容变化 → stale → 必须重批。
- **原因**：
  1. 防审批脱钩的目标正确。
  2. 批准后改错别字、补非实质说明也会整段重批，代理会反复卡死，实际会诱使绕过。
  3. 测试矩阵与实现面过大，和「修体验关节」不成比例。
  4. 「research 必须先写入 work 才算批准范围」又是新流程税。
- **意见（分级）**：
  - **v0.9 最小 basis 应包含**：feature id、level、profile、排序后 risk labels、派生 gates、code-review obligation、protected roots、已注册关键资产的路径集合（+ 可选内容 hash）。
  - **实质变更**（目标/非目标/回撤/风险相关）才要求重批；文案润色应可走显式 `rebind-approval --reason`，禁止静默改 hash。
  - 不要对每一个实现后证据（code-review/verification）入 hash（计划已排除，这点同意）。

---

### 4. implementation_approval 禁止 skipped 一刀切（Task 1.6）—— 应分路线

- **计划现状**：implementation approval 只接受 `confirmed`，`skipped` 不得生成 approved authorization。
- **原因**：
  1. 当前协议允许「跳过并接受风险」。
  2. 标准 L / 高后果任务禁止 skipped 合理。
  3. risk-minimal 或用户明确接受风险时完全禁止 skipped 过刚，会逼旁路。
- **意见**：
  - 标准 M/L，或命中高后果标签：implementation_approval **禁止 skipped**。
  - risk-minimal：允许 `skipped`，但必须写入 evidence + 风险接受记录。
  - 不要全局禁止 skipped。

---

### 5. Bash 通用白名单 hardening（Task 4）—— 降级或拆出版本

- **计划现状**：pending 时严格白名单；禁 `>`/`>>`、`$()`、`&&`、多数 pipe；未识别即 ask/deny；作为 v0.9 必达。
- **原因**：
  1. 体验**没有**报告「用 Bash 绕过待审批写业务」。
  2. 这是安全专题大项，不是体验修复主线。
  3. 过严白名单会误伤分级探索、doctor、常见只读管道，增加代理空转。
  4. 计划一边说不做通用 shell 副作用分析，一边维护长白名单，实际仍是半套策略引擎，难维护。
- **意见**：
  - **不要**作为 0.9.0 必达。
  - 若做最小补丁：只拦高置信写入（重定向到 protected roots、`rm/cp/mv` 进业务目录等）；只读探索宽放。
  - 完整 Bash 守卫拆到 0.9.1 或独立安全专题。

---

### 6. manual-test 七列强制升级八列（Task 5）—— 方向对，迁移策略改

- **计划现状**：七列→八列；旧七列在 v0.9 活动任务直接 FAIL；强制 method 枚举；static-review 不能当 manual method。
- **原因**：
  1. 「不能用静态审查冒充已运行验证」正确（体验阶段 7）。
  2. 体验要的是诚实标注 + 报错可发现（O-05/O-18），不是再做一次 schema 断裂式迁移。
  3. `partial-verification.md` 刚稳定七列，再强制八列会扩大模板/习惯断裂面，偏膨胀。
- **意见**：
  - 禁止 `static-review` / 纯 typecheck 作为 manual passed —— **保留**。
  - **优先兼容**：保留七列；用 frontmatter 可选 `method`，或实测/证据列约定前缀。
  - behavior full 时缺 method/证据再 FAIL；不要所有旧七列无差别迁移错误。
  - 自动化测试证据写 verification，并通过 REQ/MT 关联说明替代关系 —— 同意，但要写清混合场景记账规则。

---

### 7. 抬高 critical / irreversible 最低门禁（Task 6）—— 违反小任务不过度治理

- **计划现状**：
  - `critical_correctness` → plan_review full + behavior full + code-review full
  - `irreversible_consequence` → plan_review full + rollback full + behavior full + code-review full
  - 任一风险标签 → code-review 至少 light（含 risk-minimal）
- **原因**：
  1. 体验结论是：标准 L 默认门禁已 ≥ 多数单标签增量；需要的是 INFO，不是全面加码。
  2. risk-minimal 设计本意是 XS/S + 风险标签走**最小**卡，不是完整 plan-review。
  3. 若一行关键文案修复打上 `critical_correctness`，就被拖进 plan-review full，等于小任务被 L 化。
- **意见**：
  - 任一风险标签 → code-review light：**可接受**。
  - 标签无增量 → INFO：**必须保留**。
  - critical / irreversible 在 **standard M/L**：可提高 behavior（及必要时 rollback），plan-review 仍按**规模**取 light/full，不要仅凭标签强制 plan-review full。
  - 同上在 **risk-minimal**：保持 light 证据 + implementation_approval + 匹配的 behavior；**不要** plan-review full。
  - 高后果 = 标签与路线取 max，不要把开放式兜底标签变成「隐形 L」。

---

### 8. seal 删除 status 的重型事务机（Task 7 实现）—— 目标留、实现改轻

- **计划现状**：seal 事务备份、注入 cleanup_inventory、closed auth、原子删除 status 与 check-ok；seal 后业务指纹变只能开新任务、禁止 reopen。
- **原因**：
  1. 「逻辑完成 ≠ 资产清理」目标正确。
  2. 删 status + 禁止 reopen 运维过尖：漏修一行也要新 feature，成本高，诱使绕过。
  3. 事务面过大，回归风险高，不像体验修复，像状态机重写。
- **意见（更简单模型）**：

  ```text
  complete-verification → feature-check --finish
    → 写 completion（outcome + fingerprint）
    → 标记 logic-complete（status 字段或 runtime stamp，可保留 status 副本）
    → finish-guard：logic-complete + 指纹新鲜 → 允许 Git
    → finalizer 可选；not now 不堵 Git
  ```

  - 指纹漂移：阻断 Git，要求 re-verify / amend-completion；**允许同 feature 补验证**，不要强制全新 feature id。
  - cleanup_inventory 可在**第一次 finalize dry-run** 生成，不必强行塞进 seal 时的 completion schema 2。
  - 若坚持 seal：可以 closed auth，但慎删 status；至少保留只读快照便于审计。

---

### 9. 将 O-11/O-12（反向路径、REQ 字面）标为不做 —— 不同意

- **计划现状**：§1.3 / Task 9.10 认为 plan-review/code-review 已覆盖，不再增加逆向路径与 REQ 对照提示。
- **原因**：体验阶段 6 明确：plan-review 的 HIGH/CRITICAL 偏正向约束；独立 code-review 抓到的 6 个 HIGH 集中在失败路径与需求字面时机（如 H1、H5）。说明现有提示**不够**，不是「已覆盖所以别写」。
- **意见**：
  - 在 writing-plans / plan-review / code-review 的 skill 内各加 **3–5 行 bullet** 即可。
  - 不新增 gate、不进 contract、不写长文专章。
  - 「不复制到多处」同意；「完全不做」不同意。

---

### 10. 单版本 12 Task 一次发 0.9 —— 交付形态不合理

- **计划现状**：Task 1–12 串行 TDD + 提交，覆盖 contract/status/validate/hooks/finalize/upgrade/doctor/全量文档。
- **原因**：
  1. 对规则+脚本强耦合系统，一次火车易出现半升级与回归黑洞。
  2. 体验 P0 会被大重构淹没，最后 P0 没稳、新状态机又引入新摩擦。
  3. 与「小步修关节、规则不膨胀」不符。
- **意见：拆批**

| 批次 | 内容 | 建议版本 |
|------|------|----------|
| A | untracked 保护；dry-run 去向；完成与 compact 解耦（轻量）；stage-aware init；报错/help 可行动；外部确认文案；risk-label INFO；manual 证据诚实（兼容）；skill 反向路径/REQ 字面短提示；升级拒绝活动任务 | **0.9.0** |
| B | 简化版 approval basis；Edit/Write 与授权一致性（Bash 最小写入拦截可选） | **0.9.1** |
| C | 高后果门禁微调（勿把 risk-minimal 抬成 L）；事实源收敛/doctor 增强 | 小补丁或 0.9.x |
| 明确延后/砍掉 | 强制 work.md 七章；Bash 大白名单必达；seal 删 status 重型事务；critical→plan_review full（对 XS risk-minimal） | 单独 RFC |

## 四、建议的合理 0.9.0 范围

### 纳入

1. stage-aware init / approval / finish
2. finalizer dry-run 去向 + **untracked 删除硬护栏**
3. 逻辑完成与 compact 解耦（轻量 logic-complete / 慎删 status）
4. validator/help 报错可行动（指向已有 `partial-verification.md`，不写第三份格式文档）
5. 外部确认 / grillme 澄清（可降澄清成本，不可静默越 HUMAN GATE；跳过须主动告知；plan-review 不省）
6. risk-label 无增量 INFO
7. manual 证据诚实（兼容七列 + 可选 method）
8. skill 短 bullet：反向路径 + REQ 字面
9. 升级拒绝活动 status

### 明确砍掉或降级

- 强制 `work.md` 七章契约
- critical/irreversible 对 risk-minimal 强制 plan_review full
- Bash 通用白名单作为 0.9 必达
- approval basis 对全文每一个字节过敏
- seal 事务删 status + 禁止同 feature 补验证
- 以「已有覆盖」为由取消 O-11/O-12

---

## 五、改写计划时的硬约束

1. **每个改动必须能指回体验文档的具体阶段/观察/建议**；指不回的，默认降级为可选 hardening，不得挤占 0.9.0 主线。
2. **优先改脚本行为与报错，不优先改资产模型与 schema。**
3. **禁止**为防膨胀而「只加提示不修 untracked」——P0 安全问题必须硬护栏。
4. **禁止**把 L 的教训倒灌成 XS/S 或 risk-minimal 的完整 plan-review 义务。
5. **禁止**「文档标题 confirmed → 自动 skip HUMAN GATE」。
6. **单次发布以可回归、可升级、可中断为目标**；宁可分为 0.9.0 / 0.9.1，不要 12 Task 绑死一个火车。
7. 若保留 approval basis / seal / Bash 守卫，必须单独写清：**失败时用户可见的恢复动作**、以及**不会导致代理死循环**的条件。

---

## 六、一句话结论（可放在改写计划文首）

> 保留 stage-aware、完成与 finalizer 解耦、审批绑定、外部确认澄清、升级护栏；必须补上 untracked 删除硬保护；砍掉 work.md 强制统一与 risk-minimal 门禁 L 化；approval basis / Bash / seal 事务全部改轻或分批；O-11/O-12 用 skill 短提示补上。目标是修体验关节，不是重写 dev-flow 平台。
