# dev-flow v0.9 → v1.0 发布优化清单

> 状态：最终版，待实施。
>
> 目标版本：`1.0.0`。
>
> 本文是 v1.0 发布阻断清单，不收录可延后到 1.0.x 的增强项。任一清单项未完成，均不应发布 1.0。

## 1. 评审范围

本文交叉评审以下三份真实使用优化建议，并以当前 v0.9 源码和历史设计为裁决依据：

- `docs/plans/gpt5.6sol_2026-07-17-dev-flow-real-world-usage-optimization.md`
- `docs/plans/fable5_2026-07-17-dev-flow-field-usage-optimizations.md`
- `docs/grok_2026-07-17-dev-flow-optimization-recommendations.md`

源码核对范围包括：

- 路由与契约：`contract.json`、`dev-flow-policy.mjs`、`SKILL.md`、`risk-gates.md`、`standard-ml.md`
- 状态与验证：`dev-flow-status.mjs`、`dev-flow-validate.mjs`、`dev-flow-feature-check`
- 授权与收尾：`dev-flow-gate-guard.sh`、`dev-flow-finish-guard.sh`、`finish.md`、`dev-flow-feature-finalize`
- 维护与升级：`dev-flow-doctor`、`dev-flow-upgrade`、manifest、README、用户指南和现有测试

本文不评价两次宿主项目的业务实现，只修正 dev-flow 自身的分级、路由、状态、门禁、CLI、Hook、证据、收尾和维护模型。

## 2. 1.0 不可回归的设计底线

所有优化必须同时满足以下三条；不能用其中一条抵消另一条：

1. **小任务不过度治理**：无风险 XS/S 和默认轻量 M 不因流程自身创建需求书、计划、审查报告或状态档案。
2. **高风险任务不能轻易越级**：风险披露、实现前确认、行为验证和残余风险接受必须按顺序发生；不能靠实现后补文件伪装成事前完成。
3. **规则可长期维护且不持续膨胀**：优先修 contract、policy、CLI、Hook、validator 和测试；不为单次模型错误增加新等级、平行 profile、HUMAN GATE 或长篇重复说明。

补充约束：本地 Hook 是流程护栏，不是不可绕过的安全边界。1.0 的目标是让默认正确路径可执行、越级可被发现和诚实记录，而不是宣称能识别真实操作者或阻止仓库所有者绕过。

## 3. 三份方案互评

### 3.1 共同结论

三份方案对根因判断一致：v0.9 的主要问题不是门禁数量，而是自然语言路线与机器路线没有闭合。

- 轻量 M 命中风险标签后，没有既轻量又能获得授权、保留审批和运行 feature-check 的合法路线。
- feature-check 主要验证“现在存在什么”，不能证明“前置门禁是否在实现前完成”。
- gate-guard 用整段命令字符串扫描 shell 控制符，误伤引号内的证据文本；控制命令白名单又没有覆盖完整收尾生命周期。
- light gate 理论上允许内联证据，实际仍被工具要求创建独立文件，随后 compact 又删除。
- completion、manual-test、AR 和 heading 的生成端缺少机器模板，模型只能通过 validator 串行试错。
- 规则的解释层、机器层和用户指南已经出现漂移，继续增加 prose 只会扩大维护成本。

因此，1.0 应修复“路由闭包、过程时序、控制命令、轻证据、生成端和回归测试”，而不是增加治理层级。

### 3.2 GPT-5.6 方案

值得采纳：

- 对“轻量 M + 风险”“事后合规化”“代理自行接受 residual risk”“敏感配置泄漏”和 finalized 自锁的闭环分析最完整。
- 明确提出 baseline/retrospective、两阶段风险接受、inline light gate、可恢复收尾和真实场景摩擦预算。
- 强调路线说明应由 policy 派生，避免手写第三份规则真相。

需要收敛：

- 不新增独立 `dev-flow-finish` 状态机。v1.0 先在现有 Layer 2 CLI 中提供高层入口、状态提示和确定性资产生成，再由现有 `/finish` 编排；避免新增一套并行状态解释。
- 不把所有体验建议都列为不同优先级 backlog。本文只保留发布前必须完成项。

### 3.3 Fable 方案

值得采纳：

- 对 gate-guard 根因、控制命令缺口、heading 精确匹配、doctor 误报和双日期命名的源码定位最具体。
- 对实际耗时和资产“生成后立即删除”的量化最能证明摩擦来自工具缺陷，而不是门禁本身。
- 强调生成器、可复制错误消息和 light 证据复用，符合防膨胀原则。

需要加强：

- 仅用 `retrospective` 文案或 WARN 标注事后补证不足以成为 1.0 发布门槛；必须有 baseline、审批前 diff 一致性检查和明确的 retrospective 机器状态。
- 将待用户手测继续记作 `skipped` 会混淆“尚未执行”和“明确跳过”。1.0 使用非终态 `delegated`，finish 仍然阻塞，只有真实通过或用户接受后转为 `skipped + AR`。

### 3.4 Grok 方案

值得采纳：

- “优先修洞，不加层”“失败模式只能进入契约、脚本测试或极短不变量”的维护原则最清晰。
- 对跨层共享信号、独立端到端多链路和 HUMAN GATE 硬停的分类建议可直接转成短规则与 fixture。
- 明确指出实现后批量补 gate 文档仍可 36/36 PASS，是当前最高优先级的完整性缺陷。

需要收敛：

- 不采用 `--lightweight-m` 作为新的持久路线开关。它会增加 CLI flag、状态推断和文档分支；扩展现有 `risk-minimal` 到 M 更小且更符合“风险只附加最小证明”的语义。
- 不依赖文件 mtime 或 Git 提交时间证明门禁顺序；这些信号容易受文件复制、rebase、编辑器和无关 dirty diff 影响。1.0 使用 dev-flow 自己记录的业务 diff fingerprint。

## 4. 关键分歧的最终裁决

| 议题 | 最终选择 | 拒绝的方案 | 原因 |
|------|----------|------------|------|
| 轻量 M + 风险路线 | 扩展 `risk-minimal` 支持 M，policy 派生 `risk-minimal-m` | 新 profile、`hybrid`、持久 `--lightweight-m` 路线 | 不增加第三 profile；M 的标准/轻量仍由需求清晰度决定 |
| 无风险轻量 M 授权 | 允许零 status 的 M 有限授权 | 强制 standard M status | 修复 strict 模式下轻量 M 实际不可写的问题 |
| 时序证明 | init baseline + approval 前 fingerprint 对比 + retrospective 模式 | mtime、Git 时间、只写 WARN | 由流程自身生成、可测试、无需猜测外部时间 |
| residual risk | `propose-risk` → HUMAN GATE → `accept-risk` | 单步 `accept-risk`、finish 自动 skipped | 防止代理把自己的说明当作用户接受 |
| shell 控制识别 | 独立的引号感知纯分类器 | Hook 内继续堆 case、放开任意 Bash | 可单测、可复用、不削弱业务写门禁 |
| light 证据 | status 内联；full 才报告 | 每个 light gate 一份 Markdown | 降低资产税，保留证明语义 |
| 待用户手测 | 非终态 `delegated` | `pending` 直接报错、提前写 `skipped` | 语义真实；finish 仍阻塞，不降低门禁 |
| 收尾格式 | 现有 CLI 增加 start/next/scaffold，由 `/finish` 编排 | 新增第二套 finish 状态机、继续扩散独立动词 | 修生成端，不复制 validator 和 finalizer 逻辑 |
| 路线事实源 | contract + policy 为机器权威，生成机器路线矩阵 | README/SKILL/guide 手写三份 gate 表 | 减少漂移；说明文档只解释含义 |
| review 命名 | `<feature-id>-<kind>.md`，读取兼容旧双日期 | 继续前置日期 | feature-id 已默认含日期，避免重复 |

## 5. v1.0 目标架构

### 5.1 分层职责

| 层 | 权威与职责 | 1.0 约束 |
|----|------------|----------|
| Contract | 版本、schema、等级、风险标签、最小门禁、profile 合法组合 | 不承载流程散文 |
| Policy | 从 level/profile/labels/轻量选择派生 route、human gates、risk gates、code-review 和 process obligations | 所有 CLI/validator 共用，不复制判断 |
| Status CLI | 创建和原子更新 status/auth，生成确定性骨架，输出唯一下一步 | 不调用 feature-check/finalizer |
| Validator | 校验 schema、路线、证据、时序、partial 和最终资产一致性 | 不写 status、stamp 或资产 |
| Hooks | 调用统一命令分类器，根据授权状态拦截受保护写入 | 不重新实现 route policy |
| Feature check/finalizer | 新鲜度、check-ok、最终资产和可恢复清理 | 不修改 status gate，不替代 HUMAN GATE |
| Skill/commands/docs | 分类、编排和用户解释 | 不复制机器 gate 表，不新增平行规则源 |

### 5.2 版本与 schema 边界

1. `workflow_version` 升为 `1.0.0`。
2. status schema 升级，用于保存 `process` 基线/模式和 inline/report gate evidence。
3. completion schema 升级，用于长期保存 `process_mode` 与 retrospective 说明，防止 compact 后丢失过程真实性。
4. authorization schema 是否升级由实现时的字段变化决定；若新增 process/baseline 绑定字段则必须升级，不允许无版本扩展关键语义。
5. 1.0 upgrade 继续对任意活动 `status.md` 零修改拒绝；不原地猜测迁移在途 gate。
6. 已完成的 0.9 completion 继续走 legacy finalized 兼容读取，不批量改写历史资产。

## 6. 1.0 发布阻断优化项

以下全部为 must-have，不再分 P0/P1/P2。

### V1-01：补齐轻量 M 的机器路由与授权闭包

#### 问题

当前 `contract.json` 只允许 `risk-minimal` 使用 XS/S；policy 又把任何有 status 的 M 解释为 standard M。结果是：

- 无风险轻量 M 在 strict 模式下没有零资产授权入口；
- M + 风险要么无 status 越过，要么被迫补 standard M 的 requirement/writing-plans/plan-review；
- SKILL、指南和机器 policy 对“携带风险的轻量 M”给出互相矛盾的答案。

#### 必须实现

1. `risk-minimal.levels` 扩展为 XS/S/M；normal 仍要求非空风险标签，仅 `in-scope` retrospective 允许无风险 XS/S/light M 复用该 status 路线。
2. `profile=risk-minimal + level=M` 派生唯一机器路线 `risk-minimal-m`。
3. `risk-minimal-m` 只在需求边界清楚、无方案取舍、共享契约稳定时使用。
4. 必需义务：status、风险卡、标签派生的实现前 gate、`implementation_approval`、code-review、行为验证和 feature-check。
5. 明确不要求：requirement confirmation、需求书、writing-plans、requirements coverage、plan-review。
6. `authorize --level` 显式支持 `XS|S|M`；M 仅允许无风险、零 status 且当前无 active status-backed authorization 时的有限 `classified` 授权，不得为了 strict Hook 被迫初始化 standard M。
7. M + 任意风险标签必须拒绝 `authorize`，并给出 `start ...` 进入 `risk-minimal-m` 或 standard M 的可复制命令。
8. standard M 保持不变：需求/契约/方案不确定时仍走需求固化、计划和 plan-review。

这是一种刻意的不对称：轻量 L 表达“规模已跨层，但设计自明”，所以仍使用 standard profile 的 lightweight-L 路线；`risk-minimal-m` 表达“规模中等、需求清楚，只需风险最小证明”，所以复用 risk-minimal。两者不能为了命名对称而合并成新 profile 或共同 flag。

#### 主要落点

- `.claude/skills/dev-flow/contract.json`
- `.claude/skills/dev-flow/scripts/dev-flow-policy.mjs`
- `.claude/skills/dev-flow/scripts/dev-flow-status.mjs`
- `.claude/skills/dev-flow/scripts/dev-flow-validate.mjs`
- `.claude/skills/dev-flow/SKILL.md`
- `risk-gates.md`、`status-cli.md`、README、用户指南

#### 验收

- M + security、边界清楚时一次 start 成功，route 为 `risk-minimal-m`。
- 无 writing-plans/plan-review 资产仍可在审批、审查和验证完整后 finish。
- M + security、需求不清时仍为 standard M。
- 无风险轻量 M 不创建 status，strict 模式可写受保护路径。
- `authorize --level M` 携带或隐含任意风险标签时失败并指向 `start`。
- risk-minimal XS/S 和 standard M/L 既有义务不回归。

#### 防膨胀约束

- 不新增 profile。
- 不按风险标签创建不同路线。
- 不新增 HUMAN GATE。

### V1-02：让分类在影响面闭合后停止

#### 问题

当前说明已经强调拓扑，但真实执行仍会因“文件少、单模块、外部契约不变”过早判 S，直到用户追问才发现多个状态生产者/消费者和多条用户链路。

#### 必须实现

分类探查增加三个短、可停止的强信号：

1. 同一行为是否跨两个及以上架构边界，例如 middleware、API、store、UI。
2. 是否新增或修改跨边界共享信号，例如 cookie、header、响应字段、事件或协议字段。
3. 同一功能边界内是否有两个及以上行为分支、状态变体或消费者必须保持一致，例如匿名登录、真实登录、OAuth。

任两项为 true 时至少为 M；这里的“行为分支/状态变体”不等于 topology 枚举 `multi-chain`。`multi-chain` 专指多个独立端到端链路跨层传播，并要求链路间一致性、部分成功处理或协调恢复，命中时属于 L。登录/鉴权、计费或外部协议替换还应显式检查状态写入点、读取点和清理/结束点。探查只扩展一跳；证据已表明局部且契约稳定时立即停止，不建设全仓调用图。

`start` 对明显冲突的输入提供可行动提示，例如 `level=S + topology=shared-contract` 必须 WARN 或拒绝，不能静默接受明显低估。

#### 验收

- 匿名登录 fixture 稳定得到 M + security，不依赖用户二次纠正。
- 外部协议同时修改签名、session、end/error 语义时至少得到 M。
- 同一鉴权功能内的匿名/真实/OAuth 变体可判 M + shared-contract；只有形成独立跨层端到端链路时才使用 `multi-chain` 并升 L。
- 文案、局部样式、私有纯函数修复仍为 XS/S，不触发额外搜索或 status。

#### 防膨胀约束

- 只在 SKILL 分类段增加短不变量，详细案例进入测试，不新增分类文档或评分表。

### V1-03：建立可机器验证的前置时序与 retrospective 恢复

#### 问题

v0.9 能绑定 approval basis，却不能证明 status、计划和审批早于业务实现。实现后 init 并补齐文件仍可能被 validator 当作正常顺序通过。

#### 必须实现

status 新增机器生成的 `process` 块，至少包含：

```yaml
process:
  mode: "normal|retrospective"
  started_at: "<ISO-8601>"
  baseline_business_diff_fingerprint: "<git-hash>"
  existing_diff: "clean|unrelated|in-scope"
  reason: "none|<non-empty-reason>"
```

规则：

1. baseline、approval 前比较、complete-verification、check-ok 和 completion 全部复用现有 `dev-flow-fingerprint` 单一 hash 原语；不得新增第二个 manifest/hash 算法或平行 fingerprint 实现。
2. `start/init` 调用 `dev-flow-fingerprint`，把结果写入 `baseline_business_diff_fingerprint`。该脚本继续负责 staged、unstaged、untracked 业务 diff，并排除 dev-flow 自有资产；若 OpenSpec 等受管流程资产尚未纳入排除范围，只在这一现有实现中补齐。
3. fingerprint 范围不能只依赖 `protected_write_roots`，否则不完整的项目适配会造成业务改动盲区。算法或排除规则发生不兼容变化时通过 `workflow_version` 升级管理，不增加独立 `fingerprint_version`。
4. 若启动时已有业务 diff，CLI 只显示路径与 Git 状态，不显示文件内容，并要求明确声明：`unrelated` 或 `in-scope`。
5. `unrelated` 必须提供非空原因；现有 baseline hash 就是完整快照，确认 implementation approval 前重新计算的 hash 必须完全相等。路径摘要只用于用户解释或诊断，可放 runtime/evidence，不进入新的长期 schema。
6. `in-scope`：进入 `retrospective`，不得再生成或宣称事前 writing-plans、plan-review 或 implementation approval 已发生。
7. normal 模式仅在 implementation approval 首次从 pending 转 confirmed 时比较 fingerprint；漂移必须零修改失败并给出 `mark-retrospective`，确认成功时把 baseline 纳入现有 approval basis；批准后的 activate/re-confirm 不再与 B0 比较。
8. `mark-retrospective` 只切换 process mode 并记录原因，不确认任何 HUMAN GATE；之后仍使用现有 `confirm-human implementation_approval`，CLI 根据 `process.mode` 将 evidence 解释为“用户允许继续审查、修复和验证”，不得解释为追认过去的实现。
9. retrospective 不补做或完成 writing-plans/plan-review 等事前 process gate；任何 `in-scope`（含无风险 XS/S/light M）都必须有 status，并完成至少 light code-review、标签派生的风险 gate、行为验证和 feature-check。
10. completion 复用已有 `business_diff_fingerprint` 保存最终新鲜度，只新增长期 `process_mode` 与 retrospective reason/evidence；compact 后不能伪装成 normal。
11. retrospective 不等于验证 partial：功能若被充分验证仍可记录 `outcome: verified`，但过程合规状态必须保持 retrospective。

最小状态转换：

| 步骤 | normal | retrospective |
|------|--------|---------------|
| start 时已有 in-scope diff | 不允许静默归为 normal | `mode=retrospective`，记录原因与基线 |
| writing-plans / plan-review | 按 standard 路线可作为事前 gate | 禁止登记为事前完成；已有材料只能作为恢复审查输入 |
| implementation approval | 必须早于业务 diff 变化 | 同一 gate 记录“允许继续审查/修复/验证”，不追认过去实现 |
| code-review / risk / behavior | 按路线要求 | 全部按恢复后的实际 diff 执行，不可豁免 |
| completion outcome | `verified|partial` | 仍可 `verified|partial`，但 `process_mode` 永远是 retrospective |

`unrelated` 的 baseline hash 只能证明现有业务 diff 之后没有变化，不能判断它们在业务语义上是否真的与当前 feature 无关。因此路径摘要和原因必须进入用户可见的风险卡/确认上下文；本地工具不得把该机制宣传成身份认证或语义判定。

#### 主要落点

- `dev-flow-status.mjs`
- `dev-flow-validate.mjs`
- `dev-flow-fingerprint`
- `protocol.md`、`status-cli.md`
- completion 生成与校验

#### 验收

- clean start → approval 前业务 diff 变化：确认失败。
- 启动时已有无关 dirty diff，声明 unrelated 且原因非空后可继续；approval 前同一 `dev-flow-fingerprint` 发生任何变化都会转入 retrospective。
- 实现后才 init，声明 in-scope 后不能补 plan-review 伪装正常，但可在用户确认后诚实走恢复收尾。
- final completion 明确区分 normal/retrospective，并继续用已有 `business_diff_fingerprint` 证明最终新鲜度。

#### 防膨胀约束

- 不保存完整事件日志。
- 不用 mtime、Git commit 时间或远程审批服务。
- 不增加第三个人工 gate。

### V1-04：禁止代理单步接受 residual risk

#### 问题

当前 `accept-risk` 只验证非空字符串；代理可以自行把待用户执行的验证改为 skipped，并把自己的解释写成“用户证据”。

#### 必须实现

1. 新增 `propose-risk`：创建待确认 AR、绑定 feature、manual step、reason 和当前 fingerprint，生成一次性 proposal token。
2. `propose-risk` 复用现有 `[HANDOFF]`，写入 `Current gate: verification-before-completion`、`Next skill: verification-before-completion`、proposal/AR 输入、`Auto-continue: no` 和明确的 residual-risk `Stop reason`；不得新增 `[RISK ACCEPTANCE]` 或其它方括号协议形状。
3. `accept-risk` 必须引用尚未消费且 fingerprint 未漂移的 proposal token，并原样记录用户后续回复证据。只有明确表达“接受该项残余风险并继续/收尾”的回复可消费 token；模糊的“继续”“完成吧”“行吧”不可接受。
4. `/finish` 遇到 `pending`/`delegated` 手测只能提示继续验证或 propose risk，不能自动改写为 skipped。
5. feature-check 要求 manual-test、accepted_risks、partial-acceptance 和 proposal/acceptance 事实一致。
6. 现有两个开发 HUMAN GATE 和三类协议形状不扩充；residual risk 只利用 `[HANDOFF]` 已有的 `Auto-continue: no` 停顿语义，不加入通用 route gate 数组。
7. proposal runtime 丢失、worktree 切换、清理 runtime 或 fingerprint 漂移时必须重新 propose；不得从 partial-acceptance 文案反推或伪造旧 token。
8. 不使用 evidence 最小长度或强制用户手抄 AR-id 作为真实性证明；短而明确的确认合法，token 和时序负责机器绑定。

#### 验收

- 没有 proposal 的 `accept-risk` 失败。
- proposal 后没有用户确认，feature-check 失败。
- proposal 后业务 diff 漂移，旧 token 失效。
- proposal runtime 丢失后只能重新 propose，旧 AR 文案不能直接被 accept。
- 模糊继续指令不能消费 proposal；明确的短确认可以。
- propose-risk 输出的是合法 `[HANDOFF]` 且 `Auto-continue: no`，当前回合停止；仓库中不出现新的 `[RISK ACCEPTANCE]` 协议。
- 用户只输入 `/finish` 不构成 residual risk acceptance。
- verified 路径不生成 AR 或 partial-acceptance 文件。

#### 防膨胀约束

- proposal 存放 runtime，不新增长期流程资产。
- 不宣称 CLI 能验证真实人类身份；它只使默认代理路径必须经过可见的两阶段停顿。

### V1-05：统一、引号感知地识别控制命令

#### 问题

gate-guard 对整段字符串扫描 `` ` ``、`|`、`;`、`>` 等字符，引号内的结论文本也会被判为 shell 控制语法。白名单又缺 feature-check/finalize/fingerprint，导致流程在 approval-pending 或 closed 状态自锁。

#### 必须实现

1. 新增一个 Layer 0 纯命令分类器，输出：`is_control`、`has_shell_control`、`command_kind`、`reason`。
2. 分类器理解单/双引号和转义；引号外的管道、重定向、拼接、后台执行和换行会取消控制身份。命令替换是例外：反引号与 `$()` 在引号外或双引号内都会取消控制身份；只有在单引号内或转义为字面量时放行。
3. 控制面固定覆盖 status、validate、policy、doctor、feature-check、feature-finalize、fingerprint、manifest 和新的纯分类器/生成入口。
4. 支持受控的裸命令、仓库相对/绝对脚本路径以及 `node|bash|sh <精确白名单脚本>`；不得因后续参数中出现脚本名获得控制身份。
5. approval-pending、closed 和 finalized 状态都允许控制命令；业务 Bash 仍按授权阻断。
6. 拒绝消息必须包含分类结果、拒绝原因和一条可复制的合法命令。
7. 文档示例不再通过 `| tail` 压缩控制命令输出；需要时由命令自身提供 `--summary`。

#### 验收

- quoted conclusion 含 `` `code` ``、`a|b`、`x > 0`、中文分号时稳定通过。
- 引号外真实 pipe/redirect/`;`/`&&` 仍失去控制身份。
- closed 后 finalized feature-check 和 doctor 可运行。
- feature-finalize 仍必须通过自身 inventory/untracked token 防护。
- Hook 测试不再通过追加 case pattern 修复单个字符串样例。

### V1-06：提供面向路线的高层 CLI 和确定性生成端

#### 问题

当前 `init` 暴露 profile、topology、evidence-result、entry-gate 等内部组合；错误一次只报告一个字段。completion、manual-test、AR 和 heading 又要求稳定 schema，却完全靠模型手写。

#### 必须实现

1. `dev-flow-status start <feature-id> ...`：一次接收 classification 已得出的 level、topology、risk labels、轻量/标准选择和 requirement state，由 policy 推导 profile/route，原子完成 init + activate 或零 status authorize。
2. `start --dry-run` 承担只读解释：输出派生路线、必需 gate、资产上限和完整可复制的正式 start 命令，不新增独立 `explain` 动词。
3. `dev-flow-status next [<feature-id>]`：输出 route、process mode、authorization、当前 gate、真实阻塞和一条下一命令。
4. `dev-flow-status scaffold <feature-id> --asset <kind>`：生成 validator 同源的 rollback/full report、manual-test、partial-acceptance 或 completion 骨架；`--asset completion --refresh` 同时承担原 seal/refresh 语义，不新增独立 seal 动词。
5. completion refresh 从 status、check-ok 和 policy 填入机器字段；模型只补人类摘要，不能自由拼 frontmatter。
6. `complete-gate --heading` 归一化前导 `#` 和空白；失败时列出现有标题，不让模型猜。
7. topology 仍是 start 必填输入；非法或缺失时一次列出全部枚举、分类信号摘要和完整示例，不默认 `local`。
8. 所有参数错误一次列出全部缺失字段、合法枚举和完整可复制命令。
9. 不新增 `--from-classification` JSON 契约；skill 已完成的分类结果直接在一次 start 调用中提交，避免再维护一套输入 schema。
10. 保留低层 `init/activate/complete-gate` 供升级、测试和调试，但用户指南以 start/next/scaffold 三个高层入口为主。

#### 默认值原则

- `evidence-result` 可默认 `partial`，因为未完成全引用调查时不应伪称 verified。
- topology 不默认 `local`；它直接影响规模判断，必须由分类证据明确给出。
- standard M/L 仍要求明确 entry gate；高层命令根据需求状态派生并展示结果。

#### 验收

- 在分类探查已经产出合法 level/topology/labels 等输入后，两个真实 fixture 到 HUMAN GATE 前控制命令不超过 3 条；源码探查 Read 不计入该预算。
- 冷启动不再依靠多次非法枚举试错。
- completion frontmatter、manual-test 七列和 AR heading 格式试错为 0。
- scaffold 中的 TODO 不能通过 finish validator。

#### 防膨胀约束

- 不新增第二套 route 推导。
- 不新增独立任务系统或数据库。
- scaffold 只生成契约骨架，不自动宣称 gate 完成。
- 高层命令面固定为 start/next/scaffold；新需求优先扩展已有命令的只读 flag，而不是新增动词。

### V1-07：让 light gate 真正内联

#### 问题

`risk-gates.md` 声明 light 使用 inline evidence，但 code-review 和真实执行仍常被要求创建 path+heading 文件，最后 compact 又删除这些中间报告。

#### 必须实现

status gate evidence 统一支持：

```yaml
gate_evidence:
  code_review:
    mode: "inline"
    summary: "<meaningful conclusion>"
```

或：

```yaml
gate_evidence:
  code_review:
    mode: "report"
    path: "<repo-relative-path>"
    heading: "<heading>"
```

规则：

1. rollback/security/code-review/behavior verification 为 light 时默认 inline。
2. inline 必须有非占位结论；必要时记录命令、关键观察或 3–5 个行为步骤。security 且涉及登录/鉴权时，必须记录 `risk-gates.md` 验证矩阵中与本次改动相关的路径子集、观察和验证结论，不能用一句“安全无问题”代替。
3. light 默认 inline，但允许用户或项目主动保存 report，且不因此自动升级为 full；full 则必须 report，并登记为对应 kind 的资产。
4. 只有用户需要离线执行时才生成 reusable `manual-test.md`。
5. `critical_correctness`、`irreversible_consequence` 的 full code-review/verification 义务保持不变。
6. severity 升级仍由 skill 显式 promote，不让 validator 扫 Markdown 猜 HIGH/CRITICAL。

#### 验收

- risk-minimal XS/S/M 正常活动期默认只有一个 status.md。
- 无离线手测时 review 文档为 0。
- light code-review 不再要求创建独立报告。
- light 主动保存 report 时仍保持 light 义务；full 始终要求已登记的独立 report。
- full gate 缺独立报告时仍失败。

### V1-08：建立敏感配置与 secret 证据边界

#### 问题

v0.9 的业务安全审查没有约束流程自身如何读取、写入和回显 `.env*`、`.dev.vars` 或 secret，真实使用中已出现 secret value 进入新文件和会话记录。

#### 必须实现

全局不变量：

> 流程资产、status evidence、命令参数、审查报告和最终报告只允许记录 secret 名称、来源类别与 `present|missing`；禁止记录、复制或转写 secret value。

配套措施：

1. 分类和验证只检查变量名、来源与是否存在，不读取或输出值。
2. validator 只扫描 dev-flow 自有资产路径和流程输出中的高置信 secret 赋值与私钥块，不全文扫描宿主业务源码；失败只报告文件路径与 key 名，不回显 value。
3. gate-guard 对真实 secret 文件写入要求用户确认；`.env.example` 等无真实值模板允许正常编辑。
4. risk evidence 的长文本支持从受管文件读取，减少 secret 或 shell 元字符进入命令行。
5. 测试 fixture 使用假 secret，断言所有输出中只出现 key name 与 present/missing。
6. 文档明确：已经进入历史会话或真实文件的 secret 需要人工轮换；工具不能自动撤销泄漏。

#### 验收

- status、risk evidence、manual-test、completion 和命令日志不出现 fixture secret value。
- 可正常验证配置存在性和运行时连接结果。
- `.env.example` 文档性修改不被误伤。
- 宿主业务源码中的示例 token 文本不会因全仓 secret 扫描产生误报，因为 validator 不扫描该范围。

### V1-09：修正手测委托、partial 与验证新鲜度语义

#### 问题

当前 manual-test 只有 passed/failed/skipped，真实“步骤已定义，等待用户执行”只能错误写成 skipped；这会诱导模型提前生成 AR 并把未决验证变成 partial。

#### 必须实现

1. 增加非终态 `delegated`，表示已委托用户执行但尚无结果。
2. current 阶段允许 delegated；finish 阶段 delegated 与 pending 一样阻塞。
3. 用户回传证据后转 passed；用户明确接受不执行的风险后，经 V1-04 转 skipped + AR。
4. passed 必须保留 observed/evidence；full behavior verification 继续要求 method 和真实 runtime evidence。
5. static review、lint、type-check 不能冒充运行时 passed。
6. accepted risk、manual step 和 partial-acceptance 保持三方严格一致。
7. proposal、acceptance、complete-verification 和 check-ok 都绑定当前 business fingerprint；diff 漂移使旧证据 stale。

#### 验收

- delegated 不被迫伪装为 skipped，也不能 finish。
- 用户真实结果可原地更新同一步，不需要新 feature。
- partial 只在存在用户确认后的 skipped + AR 时成立。
- verified completion 不得残留 accepted risk 或开放 AR。

### V1-10：收敛 `/finish` 为幂等、可恢复编排

#### 问题

v0.9 的 Layer 3 顺序正确，但模型仍要手工拼 completion、manual-test 和 AR，并在 compact 后遇到控制命令自锁或 doctor 误报。

#### 必须实现

1. `/finish` 首先调用 `next`，每次只处理一个真实阻塞点。
2. 使用 `scaffold --asset completion [--refresh]` 生成或刷新最终资产；不复制 validator schema 到 command prose。
3. pending/delegated 验证时只允许继续验证或 propose risk。
4. complete-verification → feature-check → final assets → logic-complete 的顺序不变。
5. logic-complete 后 Git 可用；compact/full 仍是可选资产维护，`not now` 不阻塞 Git。
6. finalizer dry-run 后仍必须停等精确回复，且 compact 删除 untracked 继续要求 exact token。
7. confirm/resume 幂等：重复运行不重复 gate、不重复 AR、不重复资产。
8. finalizer 成功后 finalized feature-check 不依赖业务写授权。
9. doctor 排除受 completion schema 管理的 workflow version，不再把合法 completion 当硬编码；真正受管源码中的硬编码仍失败。

#### 验收

- completion 格式不再手工反复修改。
- `/finish` 不会自行制造 partial。
- compact 后 finalized check 一次通过。
- 合法 completion 不触发 doctor workflow-version 误报。
- dry-run 与 confirm 仍不能同回合，untracked 保护不回归。

### V1-11：修正新资产命名生成端并锁定旧格式兼容

#### 问题

feature-id 已默认带日期，模板又使用 `YYYY-MM-DD-<feature-id>-...`，导致双日期；partial-acceptance 又使用另一套命名。

#### 已满足基线

v0.9 的 `isFeatureOwnedReviewName` / `is_feature_owned_review_name` 已能识别 `<feature-id>-<suffix>.md` 与 `YYYY-MM-DD-<feature-id>-<suffix>.md`。这部分不重写，只用既有测试或新增最小回归测试锁定兼容行为。

#### 验收

### V1-12：收敛机器事实源与规则准入机制

#### 问题

当前 contract、policy、SKILL、references、README 和用户指南都包含部分路线事实，已经出现“指南说轻量 M 可带风险，policy 却只允许 standard M”的漂移。

#### 必须实现

唯一权威矩阵：

| 主题 | 唯一权威 |
|------|----------|
| 版本、枚举、profile 合法组合、最低风险门禁 | `contract.json` |
| route/human gate/risk gate/code-review/process obligations | `dev-flow-policy.mjs` |
| status/completion/partial 结构合法性 | validator + 生成器测试 |
| CLI 语法与下一动作 | CLI `--help` / `start --dry-run` / `next` |
| 分类不变量和 HUMAN GATE 停顿 | `SKILL.md` |
| 用户解释与示例 | README / guide，不作为机器判定依据 |

配套规则：

1. policy 提供 JSON 与 Markdown 路线矩阵输出；README/guide 的机器 gate 摘要由它生成或由快照测试验证。
2. SKILL 只保留不变量、分类停止条件和路线摘要；standard 细节继续按需读取 reference。
3. 新增规则 prose 时必须优先删除、替换或机器生成已有重复定义；确实没有可删除内容时，在人工删除审查中解释净增长理由，不能为了形式上的“一增一删”破坏必要规则。
4. 新失败模式只允许进入：contract/policy、脚本行为与测试、或 SKILL 不超过 5 行的不变量。
5. 历史 `docs/plans/*` 只保存决策背景，不被 doctor、skill 或 CLI 当作现行规则。
6. 每次版本发布执行人工删除审查：列出已删除、合并或由机器生成替代的重复规则，并附受控规则文件的行数 diff。doctor 只执行行数预算和可确定的快照检查，不宣称能自动识别所有语义重复。

#### 验收

- 修改 route gate 只需改 contract/policy 和测试；文档快照自动更新或明确失败。
- README、guide、SKILL 与 `next` 对同一 fixture 不再给出不同路线。
- 行数预算通过；人工删除审查与行数 diff 已归档，任何净增长都有明确且不可由机器规则替代的理由。

### V1-13：固化真实场景回归与摩擦预算

#### Fixture A：匿名登录边界

输入特征：middleware + API + cookie/store + UI；匿名、真实登录和 OAuth 是同一鉴权功能内的行为分支/状态变体，共享契约稳定；`security`，需求边界清楚。

- 一个 implementation approval；
- 不需要 requirement confirmation、writing-plans 或 plan-review；

- delegated 手测使 finish 停止；

- S 私有函数修复：零流程资产；
- 无风险轻量 M：对话内短计划 + classified 授权，无 status；
- standard M：仍要求需求确认、writing-plans、plan-review 和实现确认；
- standard/轻量 L：full 行为验证和 finalization 安全约束不下降。

#### 自动化失败用例

必须覆盖：

- active status-backed authorization 被重复 start/authorize 降级；
- 无风险 in-scope XS/S/light M 越过 status，或 retrospective 补做 plan/coverage/plan-review；

## 11. 最终评审裁决

本文已完成两轮复核并形成最终实施基线：

1. `authorize --level M` 只服务当前无 active status-backed authorization 的 normal 无风险轻量 M，M + risk 必须进入 status 路线。
2. `risk-minimal-m` 与 lightweight-L 保持刻意不对称，不新增 profile 或 `--lightweight-m`。
3. M 的“同一功能内行为分支/状态变体”与 L 的 `multi-chain` 明确分离；Fixture A 固定为 M + shared-contract，独立跨层端到端链路才升 L。
4. baseline、approval、verification、check-ok 和 completion 统一复用现有 `dev-flow-fingerprint`；不新增结构化 manifest、算法版本或平行 hash 机制。
5. retrospective 继续复用 implementation approval，但只授权后续审查/修复/验证，不追认过去实现；充分验证后可保持 `outcome: verified`，同时长期标记 process mode。
6. residual-risk proposal 丢失或 stale 时必须重新 propose；模糊继续指令不能消费 token；停顿复用 `[HANDOFF]` 的 `Auto-continue: no`，不新增协议形状。
7. 高层 CLI 固定为 start/next/scaffold；`start --dry-run` 承担解释，completion refresh 合并进 scaffold。
8. light 默认 inline、可选 report，full 必须 report；security inline 必须覆盖本任务相关验证矩阵子集。
9. secret 扫描只覆盖 dev-flow 自有资产和流程输出，不全文扫描宿主源码。
10. 删除审查使用人工检查表和行数 diff，doctor 不假装能识别全部语义重复。
11. Fixture B 出现共享契约破坏、独立 multi-chain 或协调回滚时升 L，不能被 risk-minimal-m 吸住。
12. V1-11 只修改新资产生成端；validator/finalizer 已有旧双日期兼容作为基线，仅用回归测试锁定，不重写正确代码。

本文自此不再扩展评审范围；后续工作按阶段 A → D 实施并以第 10 节发布总闸验收。
