# dev-flow 减法重构设计

## 目标

在保留高风险硬门禁的前提下，把 dev-flow 从“按等级生成一组治理资产”改成“按任务规模组织执行、按真实风险启用门禁”。

成功标准：

- XS/S 普通任务不生成流程文件。
- M 普通任务默认不落盘；确需跨会话恢复时最多维护一个工作文件。
- L 或任意高风险任务只维护一个 `work.md`；只有确需用户独立执行手测时，才允许第二个 `manual-test.md`。
- 高风险任务在风险卡输出后，必须收到用户后续确认才能实现；不能用用户最初的“直接改”自动跨过门禁。
- L 任务按可验证批次执行并持续更新恢复点，不再用一份超长计划一次性跑完整个任务。
- 不再默认生成需求说明、状态文件、三份 context manifest、覆盖矩阵、计划审查、代码审查、验证报告、feature/completion 等重复资产。

## 非目标

- 不弱化安全、数据完整性、计费、外部协议和真实行为验证。
- 不以文件数直接决定等级。
- 不为旧版多资产格式实现长期兼容层或自动转换器。
- 不引入 hook、CI、服务端状态机、数据库或第三方依赖。

## 1. 两个独立判断

### 任务规模：XS / S / M / L

等级只决定计划深度、执行分批和恢复需求：

| 等级 | 判断依据 | 默认执行方式 |
|------|----------|--------------|
| XS | 单点、行为不变或显然、可立即验证 | 直接修改并验证 |
| S | 局部行为变化，边界清楚，单模块内可独立回滚 | 简短复述后修改并验证 |
| M | 一个功能内存在多个协同步骤，但共享契约稳定、失败范围有限 | 对话内短计划；必要时一个 `work.md` |
| L | 跨架构层传播、修改共享契约/协议、影响多条调用链、存在部分成功状态或需要协调回滚 | 一个 `work.md`，按批次执行 |

任一强信号即可把任务判为 L：

- 改动跨越三个及以上架构层，例如 Web → API → Queue → Worker → Shared。
- 修改被多个调用方消费的共享接口、协议、状态模型或基础抽象。
- 同一能力必须在多条用户链路保持一致，漏改会产生静默不一致。
- 发布、失败恢复或回滚必须跨模块协调。
- 涉及外部系统协议适配，且参数、错误、流式语义或用量语义存在差异。

文件数量只能作为调查线索，不能覆盖上述判断。

### 风险标签

风险独立决定是否触发硬门禁和加强验证：

- `security`：登录、鉴权、权限、token/session、敏感信息。
- `data`：删除、迁移、数据完整性、不可逆状态变化。
- `money`：支付、订单金额、计费调用、用量归属。
- `external`：外部协议、跨系统入口、密钥、第三方失败语义。
- `availability`：共享运行时、关键链路或广泛故障面。

任务可以是 `S + security`，也可以是没有高风险标签的 L。对用户仍先报告 XS/S/M/L，并用一句话说明触发的风险门禁，不新增一套等级名。

DeepSeek 接入属于 L：它跨 Web、API、Queue、Worker、Shared，修改共享模型调用协议并同时影响四条链路；同时命中 `external`、`money` 和 `availability`。

## 2. 单一工作资产

L 或任意高风险任务使用：

```text
<FEATURE_ROOT>/<feature-id>/work.md
```

`work.md` 是计划、进度、门禁、回撤和验证的唯一事实源，不再同时维护机器 frontmatter 与重复的人类摘要。顶部保留可被轻量脚本读取的稳定状态块：

```markdown
## State
- Level: L
- Risks: external, money, availability
- Phase: planning
- Approval: pending
- Approval evidence: none
- Verification: pending
- Accepted risks: none
- Updated: 2026-07-10
```

正文只包含：

1. `Boundary`：目标、不做范围和验收结果。
2. `Context`：真正需要恢复时重读的少量非敏感路径；禁止登记 `.env*`、凭据或整个源码清单。
3. `Batches`：3–7 个结果导向批次，每批写目标、主要范围、验证和回撤；不写逐行编辑教程。
4. `Risk and rollback`：只记录已触发风险与对应控制。
5. `Verification`：命令、行为结果、残留风险和最终审查结论。

可选的 `manual-test.md` 只在测试必须由用户或外部环境独立执行时创建。它是 `work.md` 的执行附件，不再另外生成 verification、completion 或 feature 摘要。

## 3. 硬门禁

普通 XS/S/M 不因等级自动停顿。L 也不因规模大自动要求两次确认。

任意高风险标签触发一次实现前风险卡：

```text
[HUMAN GATE:implementation_approval]
边界：...
风险：...
回撤：...
验证：...
请确认是否开始实现。
[/HUMAN GATE]
```

规则：

- 风险卡输出后当前回合停止。
- 只有用户后续明确确认，才能把 `Approval` 写为 `confirmed` 并开始实现。
- 用户最初的“直接改”只表达任务意图，不能作为尚未披露风险的接受证据。
- 用户要求跳过时，必须在风险说明后明确回复“跳过并接受风险”；记录原话后把 `Approval` 写为 `skipped`。
- 需求仍有会改变风险或实现方案的歧义时，先澄清；不再固定生成第二个 requirement gate。
- 执行中出现新的高风险标签或明显扩大边界时，暂停并重新输出风险卡；普通实现偏差由代理自行修正，不打断用户。

## 4. 执行与恢复

流程顺序：

1. 读取项目适配层和直接相关源码。
2. 判断 XS/S/M/L，再单独识别风险标签。
3. 需要 `work.md` 时先写边界、批次和风险控制。
4. 高风险任务停在一次实现前确认；普通 L 可直接按批次实施。
5. 每完成一个批次，运行该批最相关验证并立即更新 `work.md` 的批次状态。
6. 发现范围或风险升级时暂停；否则自动进入下一批，不逐批向用户请示。
7. 完成后做一次基于真实 diff 的审查和新鲜验证，把结论合并进 `work.md`。
8. 标记 `Verification: passed|partial|failed`；不再运行 finalizer 或生成压缩摘要。

恢复时只读取 `work.md` 中未完成批次、相关 Context 和当前 diff。对话中的 `[HANDOFF]` 仅在实际停顿时简短输出，不再重复完整资产列表。

## 5. 轻量机器检查

保留一个小型完成检查器，职责只限于 `work.md`：

- feature id 与路径不能穿越仓库边界。
- L 或带风险标签的任务必须存在 `work.md`。
- 带风险标签时，`Approval` 必须是 `confirmed`，或是带明确接受证据的 `skipped`。
- 所有批次必须完成。
- `Verification` 必须为 `passed`；`partial` 只有记录了明确接受风险才可收尾。
- 验证证据必须包含本次实际运行的命令或行为检查。

删除旧的 context JSONL 校验、资产归档/删除和多报告存在性检查。doctor 只检查迁移包结构、项目适配值、硬门禁规则和完成检查器本身。

## 6. 项目适配层

项目适配层只保留流程真正消费的事实：

- 项目类型与包管理器。
- `feature_root`。
- install、build、type-check、lint、test、dev 和运行时验证命令。
- Git 边界与必要的项目局部规则路径。

删除 `review_root`、`runtime_root`、`sdd_progress`、context manifest、资产 retention、标准资产表及其重复 Markdown 映射。没有配置的能力写 `none`，不生成占位资产。

## 7. 兼容与迁移

- 新任务使用精简流程。
- 已完成的旧资产保持原样，不做批量迁移。
- 正在执行的旧任务可先用现有版本完成；若必须升级，人工把仍有效的边界、未完成批次、风险和验证摘要到一个 `work.md`，不复制历史报告全文。
- 版本提升为 `0.4.0`，明确这是资产契约变更。

## 8. 验证方案

用场景回归而不是继续增加规则文档：

1. 文案修改：XS，零资产，直接验证。
2. 局部表单行为：S/M，无风险标签，零资产或至多一个短 `work.md`。
3. 一行鉴权守卫：S + `security`，必须有风险卡、后续确认和验证证据。
4. 大型纯 UI 重构：L，无高风险标签，有一个 `work.md`，不要求人工门禁。
5. DeepSeek 接入：L + `external/money/availability`，一个 `work.md`、一次实现前确认、分批验证，必要时一个手测附件。
6. 高风险任务缺确认、验证失败、部分验证无接受风险：完成检查必须失败。
7. doctor、检查器测试和仓库全量文本一致性检查通过，活动规则中不再残留旧三件套或旧完成链路。

## 预期结果

以 DeepSeek 样本为例，流程资产从 10 个、约 1071 行，收敛为一个约 100–180 行的 `work.md`，必要时附一个手测清单；仍保留 L 分级、一次不可自动跨越的高风险确认、逐批恢复点、回撤说明和真实行为验证。
