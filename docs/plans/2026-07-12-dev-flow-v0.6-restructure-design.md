# dev-flow v0.6 重构设计：契约单源 + 分层加载 + 机制层门禁

> 状态：已实施。本文记录 0.6 版重构的目标、架构决策与实际结果，是本次重构的落地记录而非待评审提案。

## 背景

0.6 之前，dev-flow 在三个设计目标上的现状：

1. **小任务不过度治理**：基本达成，但任何任务都要加载 601 行 `dev-flow` SKILL + 416 行适配层模板；risk-minimal 一行改动仍要写一份不小的 YAML 证据记录。
2. **高风险不越级**：方向对，但整条防线（风险标签自评、HUMAN GATE evidence 自写、feature-check 自己记得跑）全是模型自我申报——0.4 版"没确认就直接改代码"的事故本质就是 prompt 约束失效，没有机制层兜底。
3. **规则不膨胀**：失败中。治理层从 4984 行涨到约 7600 行；同一条规则最多复述在 19 个文件；doctor 里约 40 条 `require_pattern` 把 prose 措辞锁死，每改一条规则要连带改 N 处。

## 用户确认的四个架构决策

1. hooks 纳入默认包，以 **ask 模式**注册（而非 deny），降低误伤概率。
2. 三个 `context/{implement,review,verify}.jsonl` 清单合并进 `status.md` 的 `assets` 列表。
3. status schema **v3 干净切换**，不保留 v1/v2 兼容或双写。
4. 孪生技能全部合并：`grilling`→`grillme`（保留 grillme 名）、`requesting-code-review`→`code-review`、`subagent-driven-development`→`executing-plans`（作为子模式）。

## 保留的特色链路

0.4 回滚的教训是门禁不能被静默跨过。0.6 版原样保留：

```
req-probe/openspec 固化需求 → grillme 拷问边界 → [HUMAN GATE:requirement_confirmation]（必停）
→ writing-plans → requirements-coverage → plan-review → [HUMAN GATE:implementation_approval]（必停）
→ 实现（executing-plans，可选子代理模式）→ rollback-units 审计 → code-review
→ verification-before-completion → feature-check --finish → finishing-a-development-branch
```

并新增机制层：两个 PreToolUse hooks 在业务文件被 Edit/Write 且 `implementation_approval` 仍 pending 时、或在 `git commit/push/merge` 且 feature-check 未通过时，输出 `ask` 而不是让模型自行判断是否该停。

## 最终架构

1. **单一权威契约** `.claude/skills/dev-flow/contract.json`：风险标签、最低门禁映射、gate 形态、profile、human gates、已知 gate 名、asset kinds、分类拓扑/证据结果、行数预算全部收敛于此。`dev-flow-validate.mjs` 和 `dev-flow-doctor` 都从这里读取，不再各自内置常量。契约类改动现在只需要触碰 `contract.json` + `risk-gates.md` 两处。
2. **dev-flow 技能分层**：`SKILL.md`（≤220 行）只放不变量、两轴分级、五条路径摘要和路由图；`references/protocol.md` 放 HUMAN GATE 硬协议、HANDOFF 格式、资产读取优先级、status.md v3 更新契约、completion.md frontmatter；`references/standard-ml.md` 放标准 M/L 详细编排；`references/risk-gates.md` 放风险标签定义和登录/SSO 验证清单。XS/S 不再为了分级而付出这些 reference 的上下文成本。
3. **status.md v3**：frontmatter 单写，字段从 v2 的双写对账（`Base SHA`/`Head SHA`/`Working tree dirty`/`Diff stat hash` 等）精简为 `validation.last_at`/`validation.commands`/`business_diff_fingerprint`；三个 `context/*.jsonl` 合并为一个 `assets: [{path, kind}]` 列表。
4. **Hooks（新增机制层）**：`dev-flow-gate-guard.sh` 在业务区文件被改且存在 pending `implementation_approval` 时 ask；`dev-flow-finish-guard.sh` 在 `git commit/push/merge` 且 feature-check 未留下有效戳记时 ask。两者都是"未 onboard 或无进行中 feature 时直接放行"，不误伤 XS/S。
5. **技能合并与瘦身**：3 组孪生技能合并；其余 9 个技能瘦身，公共的适配层读取、HANDOFF 示例、status 更新细则统一收进 `protocol.md`，各技能只留方法论 + 自身阻塞条件 + 一行指向 `protocol.md`。
6. **Agents 裁剪**：9 个精简为 3 个（`security-reviewer`、`code-reviewer`、`build-error-resolver`），删除的 6 个（`planner`、`architect`、`refactor-cleaner`、`doc-updater`、`e2e-runner`、`tdd-guide`）在本仓库没有结构性引用，且部分本就自述"当前项目未启用"。
7. **适配层模板重构**（416→216 行）：只保留宿主项目事实（路径别名、feature-id、项目能力、验证矩阵、测试/OpenSpec 策略、版本边界），status.md/风险门禁/completion 模板全部改为指向 `contract.json`/`protocol.md`/`risk-gates.md`；新增可选 `label_hints`（路径→建议风险标签，onboarding 时生成，只作提示不强制）。
8. **doctor 重写**（三类检查）：结构（关键文件、脚本可执行、契约可解析、孪生技能已清除、hooks 已注册）、一致性（risk-gates.md 与 contract 风险标签同步、行数预算、`workflow_version` 单源、旧项目泄漏、审查 prompt 激励）、适配层（占位符残留、test 命令疑似 dev/watch）+ 核心不变量（只指向 `protocol.md`/`SKILL.md` 两个文件，不再散射到下游技能）。

## 文档整理

- `docs/claude-dev-flow-migration.md` + `docs/claude-dev-flow-post-migration-usage.md` 合并为 `docs/claude-dev-flow-guide.md`，两份源文件已删除（git 历史可恢复）。
- `docs/claude-dev-flow-smoke-test.md` 场景重写，所有 `context manifest`/`context/*.jsonl`/`Risk Gates 表`/`last_validation_commands` 等 v2 措辞替换为 v3 的 `assets` 列表、`risk_gates`/`risk_evidence` frontmatter 字段、`validation.commands`。
- `docs/plans/` 下 5 份既有设计文档补加状态行（已实施 / 已回滚(0.4)），正文不改写，作为历史记录保留。
- `README.md`、`templates/CLAUDE.dev-flow-snippet.md` 同步移除"上下文清单"/"三件套"措辞，补充 hooks 机制层说明。

## 验收结果

| 验收项 | 结果 |
|---|---|
| 风险标签不抬高规模等级 | 通过：`risk-gates.md` 与 `SKILL.md` 均声明风险标签独立于规模轴；`dev-flow-validate.mjs` 不因 `risk_labels` 非空修改 `level` |
| 两个 HUMAN GATE 必须停轮 | 通过：`protocol.md` 硬协议 + `dev-flow-gate-guard.sh` 机制层兜底，`dev-flow-hooks-test` 覆盖 pending/confirmed 两种场景 |
| grillme 在写计划前、plan-review 在实现前、code-review 不能替代 plan-review | 通过：`protocol.md`「HANDOFF 格式与路由表」明确顺序；`SKILL.md` 与 `protocol.md` 均重申 code-review 不能倒填为计划门禁 |
| 风险标签最低门禁不可降为 none | 通过：`dev-flow-feature-check-test` 含 `security_review` 被改为 `none` 时必须 FAIL 的用例 |
| feature-check --finish 仍强制 | 通过：未改动该约束 |
| hooks 只加护栏、不减既有校验 | 通过：`dev-flow-doctor` 静态检查项相对重写前只增未减；hooks 是新增的独立检查层 |
| dev-flow SKILL.md ≤220 行 | 通过：112 行 |
| 契约改动只需碰 contract.json + risk-gates.md | 通过：风险标签数据全部来自 `contract.json`，`risk-gates.md` 由 doctor 校验同步 |

## 行数报告

以 `.claude/` + `templates/`（不含 `docs/`、`.claude/runtime/`、`.claude/settings.local.json`）为口径，重构前基线取 HEAD（`0fcc43c`，本次重构开始前的最后一次提交）：

| 分类 | 重构前 | 重构后 | 变化 |
|---|---|---|---|
| prose/规则层（SKILL.md、references、rules、commands、agents 描述） | 5601 | 3142 | **-44%** |
| 脚本/测试/配置层（scripts、hooks、contract.json、settings.json、agents/*.yaml） | 1864 | 2323 | +25% |
| **合计** | **7465** | **5465** | **-27%** |

验收基线原定"总行数 ≤4800"未达成（实际 5465）。差额完全来自脚本/测试层的净增长，具体是：

- 新增 hooks 机制层：`dev-flow-gate-guard.sh`（101 行）+ `dev-flow-finish-guard.sh`（93 行）+ `dev-flow-hooks-test`（137 行）= 331 行，这是决策①（hooks 纳入默认包）的直接成本，是本次重构对目标②（高风险不能越级）最核心的补强。
- `dev-flow-feature-check-test` 为覆盖 v3 的 `assets` kind 校验、风险标签负例、戳记过期等新场景从 311 行增至 416 行——新增的是回归保护用例，不是重复内容。
- 新增 `contract.json`（65 行）和 `dev-flow-fingerprint`（52 行）：前者是本次重构本身要求的单一契约源，后者是从 `dev-flow-validate.mjs` 抽出的可复用指纹计算逻辑。
- 抵消项：`dev-flow-feature-check` 387→237 行（-150），删除 JSONL manifest 循环逻辑。

prose/规则层的下降幅度（-44%）远超原定标准，是目标③（规则不因持续演进而膨胀）实际要解决的部分；脚本/测试层的增长是目标②要求的机制化直接代价。未对脚本层做进一步压缩，因为可压缩的部分只剩下删测试用例或简化 hook 判断逻辑——两者都会削弱而不是巩固高风险门禁，与本次重构的初衷相反。是否接受"总行数以此为准"或要求进一步压缩，留给用户确认。

## 风险与后续

- 技能合并后的路由依赖被合并技能的触发关键词已并入目标技能 description；已用全库 `rg` 扫描确认无 `grilling/`、`requesting-code-review/`、`subagent-driven-development/`、`context/*.jsonl`、"上下文清单"、"三件套" 等悬空引用残留（历史 `docs/plans/*.md` 除外，按设计保留原文）。
- hooks 是 ask 模式护栏，不是强制安全边界；`dev-flow-hooks-test` 覆盖未 onboard、无进行中 feature、pending/confirmed 等场景，避免误伤 XS/S 或非 dev-flow 场景。
- doctor 重写后删除的是逐句锁死 prose 措辞的检查，保留和新增的是不变量、契约同步和结构性检查；行为级回归由 `docs/claude-dev-flow-smoke-test.md` 场景走查承担。
