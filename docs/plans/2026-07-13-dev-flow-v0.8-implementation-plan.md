# dev-flow v0.8 精确实施计划

计划落盘目标：`docs/plans/2026-07-13-dev-flow-v0.8-implementation-plan.md`

## 一、本轮补充结论

Grok 提出的六点都成立，正式锁定以下行为：

1. `complete-verification`、`feature-check`、`finalizer` 不能互相套壳；只由 `/finish` 负责顶层编排。
2. finalization dry-run 后必须停止，只有用户回复明确选择才允许 `--confirm`。
3. compact/full 完成后，共享 `review_root` 中不得继续遗留当前 feature 的中间报告。
4. HIGH/CRITICAL 是否触发 promote 由 review skill 判断；CLI 不扫描自然语言或自动猜严重级别。
5. v0.8 必须处理 shared SDD、旧 authorization、旧 check-ok 和各种在途 feature 状态。
6. 轻量 L 明确派生 `code-review: light`；它不是 `risk_gates` 字段，而是 finish 路线义务。

版本保持：

- `workflow_version: 0.8.0`
- `status_schema: 3`
- `completion_schema: 1`
- 新生成的 completion 必须写 `workflow_version: "0.8.0"` 并使用严格 final-assets 契约；只有无 active status 的历史 finalized completion 可省略并获得 WARN。

# 二、脚本调用架构

## 1. 分层

### Layer 0：纯策略和只读基础能力

- `dev-flow-policy.mjs`
- `dev-flow-fingerprint`

职责：

- 路线和 gate 推导。
- business diff fingerprint。
- 不修改 status、资产、stamp 或 authorization。
- 不调用任何上层脚本。

### Layer 1：只读 Validator

`dev-flow-validate.mjs` 提供：

```text
status <root> <status-path> --stage current|approval|finish

feature <root> <feature-id> --stage active|finalized

final-assets <root> <feature-path> <completion-path>
  [--status <status-path>]
```

职责：

- 校验 status、路线、资产内容、partial contract 和最终事实源。
- `feature --stage active` 不要求 feature/completion。
- `feature --stage finalized` 不依赖 status。
- 不写 status、check-ok 或业务资产。
- 只允许调用 Layer 0。

### Layer 2：单一职责脚本

#### `dev-flow-status`

- 唯一 status/authorization 写入器。
- 只调用 validator 和 fingerprint。
- 禁止调用 feature-check 或 finalizer。

#### `dev-flow-feature-check`

- 只读检查 + 写 `check-ok`。
- 调用 validator 和 fingerprint。
- 禁止调用 status CLI 或 finalizer。

#### `dev-flow-feature-finalize`

- 只负责资产 dry-run、归档、删除和事务恢复。
- 调用 validator、fingerprint，并读取现有 check-ok。
- 禁止调用 status CLI 或 feature-check。

### Layer 3：编排层

- `/finish`
- `verification-before-completion`
- `finishing-a-development-branch`

只有这一层能按顺序调用：

```text
complete-verification
→ feature-check --finish
→ 生成 final assets
→ finalizer dry-run
→ 停等用户
→ finalizer --confirm
→ feature-check --finish（finalized）
→ Git 收尾
```

## 2. 禁止调用边

实施后由 doctor 静态检查：

```text
dev-flow-status        -X-> feature-check
dev-flow-status        -X-> feature-finalize
feature-check          -X-> dev-flow-status
feature-check          -X-> feature-finalize
feature-finalize       -X-> dev-flow-status
feature-finalize       -X-> feature-check
validator/fingerprint  -X-> 任意 mutator/wrapper
```

## 3. `complete-verification` 的精确定义

`complete-verification` 执行：

1. 校验输入路径、report、manual-test 和 command。
2. 校验 code-review 已完成。
3. 计算 business fingerprint。
4. 在内存中构造 candidate status。
5. 原子写入 candidate。
6. 调用 `validate status --stage finish`。
7. 调用 `validate feature --stage active`。
8. 任一失败则恢复原 status。
9. 成功时保留 verification gate 和 validation 信息。

它明确不做：

- 不调用 `dev-flow-feature-check`。
- 不生成 check-ok。
- 不调用 finalizer。
- 不生成 feature.md/completion.md。
- 不删除任何用户报告文件。

随后必须由 `/finish` 显式运行 `feature-check --finish`；只有 feature-check 可以生成 check-ok。

# 三、路线与 evidence 默认值

| 路线 | Approval 前 | Finish 前 | Code review |
|---|---|---|---|
| 无风险 XS/S | 无 tracked gate | 项目验证 | 不强制 |
| risk-minimal XS/S | 标签触发 rollback/security + implementation approval | behavior verification | 默认不强制 |
| 轻量 M | 无 status | 项目验证 | light，但不落 status |
| 轻量 L | rollback light、标签 gate、implementation approval | behavior light、verification | `light` |
| 标准 M | requirement confirmation、writing-plans、coverage（若触发）、plan-review light、标签 gate、implementation approval | verification | `light` |
| 标准 L | requirement confirmation、writing-plans、coverage full、plan-review ≥ light、标签 gate、implementation approval | behavior full、verification | `full` |

### 轻量 L code-review

`code-review` 不加入 `risk_gates`，由路线策略派生：

```text
required: true
evidence_level: light
```

finish 必须同时存在：

```text
completed_gates:
  - code-review

gate_evidence:
  code_review:
    path: "<已有文件>"
    heading: "<唯一 heading>"
```

如果 code review 发现 CRITICAL/HIGH：

- code-review skill 创建独立 review 报告。
- 用 `complete-gate code-review --evidence-file ...` 登记该报告。
- 不需要也不允许 `promote-gate code-review`，因为 code-review 不是 contract risk gate。
- 阻塞 finding 处置完成前不得 complete gate。

标准 L 始终要求独立 code-review 报告。

# 四、`promote-gate` 职责边界

## CLI 负责

`promote-gate` 只接受 `contract.risk_gates`：

- requirements_coverage
- plan_review
- rollback_units
- security_review
- behavior_verification

CLI 只执行：

- 校验 gate 合法。
- 校验 `none → light → full` 单调提升。
- 记录用户/skill 给出的提升理由。
- 拒绝降级。
- 重新运行 validator。

## Skill 负责

以下 skill 负责识别 severity：

- requirements-coverage
- plan-review
- rollback-units
- security reviewer
- verification-before-completion

规则：

- light 执行过程中一旦产生 CRITICAL/HIGH finding，skill 必须先调用 `promote-gate ... --to full`。
- 即使 finding 随后被修复，也保留 full 报告和 disposition，便于审计。
- full 证据完成并关闭阻塞项后，再调用 complete-gate。

## 明确不做

- CLI 不扫描 Markdown 寻找 `HIGH`/`CRITICAL`。
- Validator 不根据自然语言自动改变 gate level。
- 不引入 severity parser 或新的 finding schema。
- Feature-check 只校验声明后的 gate level、证据形态和未关闭义务。

# 五、`/finish` 状态机与停等语义

## 状态 A：需要验证

条件：

- check-ok 不存在。
- check-ok fingerprint 过期。
- verification 尚未闭环。

动作：

```text
complete-verification
→ feature-check --finish
```

feature-check 成功后进入状态 B。

## 状态 B：已验证但未 finalization

动作：

1. 生成或更新精简 feature.md/completion.md。
2. `completion.md` 写入 `workflow_version: "0.8.0"`。
3. 运行 final-assets validator。
4. 读取 `dev_flow.artifacts.retention`，用该项目默认 `compact|full` 运行 finalizer dry-run。
5. 输出清单、统计和 `inventory_sha256`。
6. 输出：

```text
[ASSET FINALIZATION]
Feature: <feature-id>
Verification: verified|partial
Inventory: <sha256>
Working set: <files/bytes>
Long-term keep: <files/bytes>

Reply exactly:
- compact
- retain full
- not now
[/ASSET FINALIZATION]

Auto-continue: no
```

7. 当前回合必须立即停止。

禁止：

- 同回合自动运行 `--confirm`。
- 把用户之前的 implementation approval 当 finalization 确认。
- 把“继续”“好的”“完成吧”等模糊回复视为删除授权。
- 未收到明确选择时进入 Git 收尾。

## 状态 C：用户选择 `compact`

下一回合：

```text
dev-flow-feature-finalize <feature-id>
  --retention=compact
  --confirm
  --inventory <sha256>
```

成功后：

1. 运行 finalized feature-check。
2. 进入 Git 收尾选项。

## 状态 D：用户选择 `retain full`

使用同一 inventory：

```text
dev-flow-feature-finalize <feature-id>
  --retention=full
  --confirm
  --inventory <sha256>
```

同一批资产从“删除”改为“移动到 archive”，无需第二次选择。

## 状态 E：用户选择 `not now`

行为必须写死：

- 不调用 finalizer。
- status、feature、completion 和中间资产保持原样。
- 现有 check-ok 保留。
- 回复只能声明“验证已完成/partial，但资产尚未 finalized”。
- 禁止声明“可提交”“已收尾”。
- finish guard 对 git commit/push/merge 返回 ask，原因是未 finalized。
- doctor 输出 ready-to-finalize WARN。
- authorization 暂时保留；若之后修改业务代码，business fingerprint 变化，check-ok 自动变 stale。
- 再次运行 `/finish`：
  - fingerprint 未变时可复用验证结果。
  - fingerprint 已变时必须重新 complete-verification 和 feature-check。
  - 无论是否复用，都重新生成 dry-run 和 inventory hash。

## Inventory 防漂移

dry-run 的 `inventory_sha256` 覆盖：

- retention 无关的资产集合。
- 每个资产路径、类型、大小和内容 hash。
- feature.md/completion.md hash。
- reusable manual-test 判定。
- 当前 status hash。

confirm 时重新计算：

- hash 一致才允许变更。
- 不一致则零修改失败，要求重新 dry-run 并再次停等用户。

# 六、compact/full 与 review-root 规则

## Feature-owned review 定义

满足任一条件即属于当前 feature：

1. 已登记在当前 status.assets。
2. 被当前 gate_evidence/risk_evidence 引用。
3. 文件名精确匹配：

```text
<feature-id>-<known-suffix>.md
YYYY-MM-DD-<feature-id>-<known-suffix>.md
```

known suffix：

- plan-review
- code-review
- security-review
- verification
- manual-test
- partial-acceptance

模糊包含 feature id、其他 feature 文件、未匹配标准后缀的未登记文件均不得自动处理。

## Compact

成功后：

- feature-owned review 从 `review_root` 删除。
- reusable manual-test 移动到：
  `<FEATURE_ROOT>/<feature-id>/manual-test.md`
- partial-acceptance 的每个 accepted AR 必须先以 `## AR-xxx` + reason/evidence 长期事实段写入 completion，然后删除。
- 主 feature 目录只保留：
  - feature.md
  - completion.md
  - reusable manual-test.md
  - 既有 archive（如果过去曾明确 full）
- status、计划、需求、coverage、rollback、patches、SDD 全部退休。

## Full

成功后：

- feature-owned review 从共享 `review_root` 移走。
- 移动到：
  `<FEATURE_ROOT>/<feature-id>/archive/<timestamp>-<nonce>/reviews/`
- feature 目录中间资产移动到：
  `<FEATURE_ROOT>/<feature-id>/archive/<timestamp>-<nonce>/feature/`
- reusable manual-test 仍移到主 feature 目录。
- SDD/checkpoint 是 runtime，不进入 archive，成功后删除。
- 共享 review-root 中同样不得残留当前 feature 报告。

## Finalized 校验

compact/full 都必须断言：

- status 不存在。
- 当前 feature-owned review-root 文件数为 0。
- 主目录符合 allowlist。
- full 有新的 archive。
- compact 没有本轮产生的 archive。
- 其他 feature 的 review/feature 资产完全未变化。

# 七、在途迁移契约

## 1. Legacy shared SDD

升级前：

```text
.claude/runtime/sdd/
  task-*.md
  review-*.diff
  progress.*
```

处理：

- 不猜测其 feature 归属。
- 整个 legacy shared SDD 移入本次 upgrade backup：
  `.claude/runtime/dev-flow/upgrade-backup-<timestamp>/legacy-sdd/`
- 创建新的空 `.claude/runtime/sdd/`。
- v0.8 只读取 `<feature-id>/` 子目录。
- 在途任务恢复时从 status、计划和 git diff 重建 checkpoint。
- upgrade 失败时恢复 legacy SDD 原位置。

## 2. 旧 authorization

- 成功升级前备份。
- 成功升级后删除/失效。
- 用户恢复业务写入前必须重新运行 `activate`。
- activate 使用 v0.8 approval-stage。
- 升级失败时恢复原 authorization。

## 3. 旧 check-ok

所有：

```text
.claude/runtime/dev-flow/*.check-ok
```

处理：

- 升级时备份并失效。
- 不复用 v0.7 stamp，因为 stamp 没有 policy version。
- 下次 `/finish` 必须重新运行 feature-check。
- 升级失败时恢复。

## 4. status 存在、completion 不存在

视为 active/in-flight：

- 保留 status 和全部资产。
- 不自动补 completed gate。
- 不自动 compact。
- authorization/check-ok 仍按上述规则失效。
- 旧 `completed_gates/current_gate` 若使用 risk gate 的 snake_case 名称，恢复时先运行 `dev-flow-status repair <feature-id>` 规范化为连字符；repair 不补造 gate 事实。
- 下次 approval/finish 使用 v0.8 路线；缺失 gate 时给出补齐命令。

## 5. status 与 completion 同时存在

视为 verified-but-unfinalized：

- 两者和中间资产全部保留。
- doctor 输出 WARN。
- `/finish` 将 completion 标准化为 v0.8：
  - 添加 `workflow_version: "0.8.0"`。
  - `risk_labels` 与 `accepted_risks` 分别和 status 精确一致。
  - partial 为每个 accepted risk 补齐唯一 `## AR-xxx` reason/evidence 段；verified 保持 `accepted_risks: []` 且不写 AR 段。
  - 与 status 重新比对。
- 然后进入 dry-run，禁止升级脚本自动删除。

## 6. completion 存在、status 不存在

视为 legacy finalized：

- 完全保留，不重新 compact。
- completion 无 `workflow_version` 或版本低于 0.8 时，使用旧 schema 1 最小校验。
- 历史 partial 缺少逐 AR 长期事实段时只 WARN，不 FAIL；verified 不要求新增 heading。
- 新创建的 v0.8 completion 必须执行严格 final-assets 校验。

## 7. 已有 full archive

- 保留既有 archive。
- 升级不移动、不合并、不删除。
- 新一轮 finalization 只能创建新的时间戳目录。
- compact 不得静默删除过去明确保留的 archive。

## 8. 旧 manual-test/partial

- 保留现有 table/frontmatter 兼容解析。
- 在途 feature 下次 finish 时规范化为 MT/AR 当前契约。
- 已 finalized feature 不要求补写历史 manual-test。

# 八、执行期资产压缩

## 1. 计划

writing-plans 使用唯一任务格式：

```markdown
### Task 3: <目标>

**Requirements:** R-002, R-004
**Depends on:** Task 1
**Writes:**
- Modify `src/a.ts`
- Create `src/b.ts`

**Change:** 行为变化和接口边界。
**Verification:** 可执行验证。
**Rollback:** 回撤范围与顺序。
```

规则：

- 删除 Files/Writes 重复。
- 不预写完整实现。
- 只在消除歧义时保留短接口签名或 5–15 行骨架。
- 计划长度只做观测，不设 hard gate。

## 2. 需求

存在成熟 design 时使用 delta 卡：

- 本次范围。
- 非目标。
- 验收差异。
- 风险差异。
- 与 design 的差异。

不复制 design 原文。

## 3. Light/full

- risk-label light → `risk_evidence`。
- process-gate light → `gate_evidence` 指向已有 heading。
- full → 独立资产。
- light 不创建空报告。

## 4. Review/verification

- Review：findings + evidence + disposition + remaining risks。
- Coverage：保持短矩阵。
- manual-test：唯一 MT 步骤源。
- verification：只引用 MT ID、命令、verdict 和异常。
- status：只保存状态和指针。
- feature/completion：只保存长期事实，不复制过程材料。

# 十一、最终验收

- 无风险 XS/S 不增加治理。
- 标准 L 不能越过 skeleton。
- human skipped 合法，但不代表 process gate 完成。
- complete-verification、feature-check、finalizer 无调用环。
- `/finish` dry-run 后必停。
- not now 的 stamp、guard、doctor 行为完全确定。
- compact/full 后 review-root 不残留当前 feature 报告。
- 轻量 L 明确要求 code-review light。
- HIGH/CRITICAL promote 由 skill 触发，CLI 只执行声明。
- 所有 v0.7 在途状态都有确定迁移路径。
- 执行期计划和审查不再互相复述。
- compact 后长期资产符合 allowlist。
- `vue3.md` 从源包和目标升级项目中消失。
- 所有测试输出 `Summary: 0 failure(s)`。

## 明确不做

- 不新增 level、profile、human gate。
- 不新增 process waiver。
- 不自动解析 finding severity。
- 不静默删除资产。
- 不把 check-ok 当 finalization 完成证明。
- 不硬合并 verification/manual-test。
- 不设置行数 hard gate。
- 不强制子代理、自动 commit 或每任务持久报告。
- 不破坏已 finalized 的历史 completion。
