# Dev Flow 1.0 发布前遗漏修复设计

## 目标

封堵复审发现的授权降级、retrospective 越级、重新批准死锁、`next` 错误指令、残余风险模糊签收和 secret 写入检测漏口，同时补齐真正的 policy 组合矩阵。不新增 profile、持久化 route、fingerprint 算法或 diff manifest。

## 路由设计

- 任何 active status-backed authorization 都不得被 `start/authorize` 写成 classified；同名 status 的重复 start（含 dry-run）直接零修改拒绝。
- `in-scope` 始终进入 status-bearing retrospective。无风险 XS/S 复用 `risk-minimal`，无风险 light M 复用 `risk-minimal-m`；仅 retrospective 允许该 profile 为空风险标签。
- retrospective 始终要求 implementation approval、至少 light code-review、行为验证和 feature-check；保留 rollback/security 等实际风险门，不追补 requirement confirmation、writing-plans、requirements-coverage 或 plan-review。
- standard retrospective 仍接受 `--requirements` 作为分类输入，但不生成 requirement entry gate；因此 dry-run 与真实执行使用同一路由。

## 授权与状态设计

- `B1 === B0` 只在 implementation approval 从 pending 首次转为 confirmed 时检查。批准后的 `activate` 或重新 `confirm-human` 只重算已有 approval basis，不再比较当前业务 diff 与 B0。
- gate 完成后 `current_gate` 转向下一 gate，`next` 的 blocker 与 command 都从当前 policy 状态派生；implementation 固定输出 `/executing-plans`。
- `accept-risk` 除“接受 + 继续/收尾”外，还必须通过 AR ID 或“具名/残余/上述/刚列出/this named residual”等指代绑定当前 proposal。

## Secret 和测试设计

- Edit/Write 对顶层与子目录的 `.env*`、`*.pem`、`*.key`、`secrets.*`、`credentials.*` 一致要求确认，`.env.example` 例外。
- command classifier 递归检查 `bash|sh -c <script>` 的内层重定向和 secret 目标，仍不宣称是通用 shell 安全边界。
- policy 测试枚举 topology × level × execution × risk × process mode；集成测试覆盖每个本次发现的可复现失败。
