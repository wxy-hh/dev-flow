# dev-flow v0.7：执行闭环与升级一致性优化

## 发布边界

这是一次单一的 `v0.7.0` 发布，不向目标项目分发任何中间切片。`dev-flow-upgrade`、受管清单、status CLI、严格门禁和全部回归测试必须作为同一批变更发布，防止再次出现契约已更新、skill/doctor 仍旧版本的半升级状态。

保持 `status.md` schema v3，不增加第三种 L profile。

## 已锁定行为

### 状态、验证与 partial

- `verified`：所有必需行为步骤均为 `passed`，且验证证据新鲜。
- `partial`：无 `failed` 或 pending 步骤；每个 `skipped` 步骤均关联唯一 `AR-xxx`；用户明确接受每个风险。
- pending、空实测、`待执行`、缺风险关联或 `failed` 一律使 feature-check 失败。
- `partial` 可正常提交、推送、合并或创建 PR；但 completion、check stamp、收尾输出必须为 `outcome: partial`，展示未验证步骤和风险摘要，禁止使用“验证通过”。
- 新增 `<REVIEW_ROOT>/<feature-id>-partial-acceptance.md`，仅 partial 时由 CLI 生成，包含 `AR-xxx`、关联步骤、原因、用户原话证据、确认时间。
- 手测表固定为：`ID | 操作 | 预期 | 结果 | 实测 | 证据 | 风险 ID`。`结果` 仅允许 `passed|failed|skipped`。
- `accepted_risks` 仅保存已接受的 `AR-xxx`；feature-check 必须验证手测表、状态、partial-acceptance 三方一致。

### 写入门禁

`project-workflow.md` 新增 `enforcement_mode` 与 `protected_write_roots`。一个 worktree 同时只允许一个 `.claude/runtime/dev-flow/write-authorization.json`。

| 模式 | 无授权业务写入 | pending M/L 或 risk-minimal |
|---|---|---|
| `off` | allow | allow |
| `ask` | allow | ask |
| `strict` | deny，并提示运行 `/dev-task` 分类 | deny |

- XS/S 完成分类后由 CLI 写入有限授权；risk-minimal 仍须实现前确认。
- M/L 仅在 `implementation_approval` 已确认后更新为 `approved`。
- 白名单仅限 `.claude/runtime`、feature root、review root、OpenSpec 流程资产；不使用宽泛的 `docs/**` 白名单。
- v0.7 只强制 Edit/Write/MultiEdit/NotebookEdit；Bash 写入拦截不是本版本承诺。
- 新项目 onboarding 默认 `strict`；旧项目升级后默认 `ask`，只有显式选择才升为 `strict`。
- 严格模式是流程护栏而非安全边界；必须以真实 Claude Code fixture 验证 deny 输出生效。

### 成熟设计与覆盖

- grillme 默认最多五个阻塞决策；仅在出现架构新分支、不可逆风险或配额/成本问题时扩至八个。
- 只有已展示完整 Decision Log 与 `[HUMAN GATE:requirement_confirmation]` 后，用户的“继续”才可视为确认；其他场景只表示继续提问或继续流程。
- 成熟设计可将 coverage 内嵌在计划的 `## Requirements coverage` 章节，并使用 light gate。
- `assets` 只保存真实文件路径；禁止使用 `path#heading`。
- 新增 v3 可选 `gate_evidence.requirements_coverage`：

  ```yaml
  gate_evidence:
    requirements_coverage:
      path: "docs/features/<id>/实现计划.md"
      heading: "Requirements coverage"
  ```

  validator 必须确认文件存在、位于允许根目录且标题唯一。
- Locked Decision Log 记录决策 ID、来源、结论、影响需求与非目标。plan-review 遇到冲突只能标记 `scope-conflict` 并停下请求用户决定。

## 验收用例

- status CLI：三种 profile 的 init 通过 validator；非法参数、重复 gate、越界 asset、缺用户证据均不改变文件；repair 不得改变 pending、manual-test 或风险接受语义。
- partial：pending、failed、无 AR、AR 无接受记录、AR 与步骤不匹配均失败；完整 skipped + AR + 用户证据通过，输出和 completion 均为 partial，收尾允许正常 Git 操作但不出现 verified 文案。
- strict gate：无授权 strict 写入 deny；ask 模式 pending 写入 ask；off 始终 allow；approved M/L 与已分类 XS/S 正常写入；risk-minimal 未批准仍被拒绝；流程资产、路径穿越和 symlink 分别验证。
- 真实 Claude Code smoke fixture：实际 Edit 调用收到 deny，而不仅检查 hook JSON。
- coverage：内嵌章节的 path/heading 合法时 light gate 完成；缺标题、重复标题、锚点伪装为 asset 时失败。
- grillme：五个决策内收敛；第六项只有触发条件时出现；裸“继续”不会越过确认 gate。
- 并行：重叠 Writes 串行、无重叠并行、Phase 0 或批次集成失败阻止后续派发、untracked 文件进入审查包。
- upgrade：使用“v0.6 contract + 旧 skill/doctor”的混搭 fixture；check 报漂移，apply 后 preflight 通过、废弃目录清理、项目适配层/自定义 skill 保留；注入失败后所有受管文件回滚。
- 完整回归：更新 hooks、feature-check 既有测试，并新增 status、upgrade 与“本次真实会话复现”端到端 fixture。

## 实现前补钉（开干前锁定）

以下条目覆盖补充版设计审查中仍会卡住实现或验收的缺口；与上文冲突时以本节为准。

### 1. 路径拦截模型（模型 A）

采用 **默认放行 + 命中才拦**，不采用「除白名单外全部 deny」。

- **`protected_write_roots`**：业务路径 glob 列表（onboarding 填写，如 `apps/**`、`src/**`、`packages/**`）。仅当写入目标命中其中任一项时，才套用 enforcement 表。
- **永远不拦（流程白名单）**：
  - `.claude/**`（含 skills / hooks / commands / rules / runtime）
  - `<feature_root>/**`
  - `<review_root>/**`
  - `openspec/**`（若项目启用）
- **未命中** `protected_write_roots` 且不在白名单内的路径：allow（避免漏配 roots 时误伤杂项文件）。
- **`enforcement_mode: strict` 且 `protected_write_roots` 为空或含不安全项**：`dev-flow-doctor --preflight` **FAIL**，不得宣称已启用 strict 门禁。
- 无 `project-workflow.md`（未 onboarding）：与现网一致，gate-guard **静默 allow**。

### 2. `write-authorization.json` 与 XS/S 分流

一个 worktree 仅一份 `.claude/runtime/dev-flow/write-authorization.json`。字段保持上文约定；**读写业务受保护路径时只认这一份授权**，不再扫描全部 `status.md` 的 pending 门禁（消除幽灵 feature 误拦）。

| 场景 | CLI | 是否创建 `status.md` | authorization `state` |
|------|-----|----------------------|------------------------|
| 无风险 XS/S | `dev-flow-status authorize --level XS\|S [--note ...]` | **否** | `classified`（有限业务写） |
| 风险 XS/S | `init ... --profile risk-minimal` + 标签 | **是** | `approval-pending` → 确认后 `approved` |
| 标准/轻量 M/L | `init ... --profile standard` | **是** | `approval-pending` → `implementation_approval` 确认后 `approved` |

补充规则：

- `authorize`（无风险 XS/S）**不得**接受 `--profile standard`；无标签时也不得写入 risk-minimal status。
- `activate <feature-id>` 切换活动功能前：将旧授权标为 `closed`（或覆盖并记录）；**v0.7 不支持**同一 worktree 并行两个进行中的 feature 授权。
- `state: closed` 或文件缺失：受保护路径按「无授权」行处理（strict → deny + 提示）。
- deny / ask 文案必须带可复制下一步（例如先 `/dev-task` 分类，或 `authorize` / `init` + `confirm-human`），避免模型空转。
- Path Card / 分类完成后：**同回合内**写入授权（XS/S 用 `authorize`，M/L 用 `init`+`activate`），不得把「拿到写权限」推到用户下一轮才完成。

CLI 表面在上文基础上增加：

```text
dev-flow-status authorize --level <XS|S> [--note <text>]
```

### 3. `gate_evidence` 属于 schema v3 可选字段

- **不升 v4**。`contract.json` / `protocol.md` 声明 v3 允许可选块 `gate_evidence`。
- 旧 status 无该块：仍然合法。
- `requirements_coverage: light` 且证据在计划内嵌章节时：`complete-gate requirements-coverage` **必须**写入：

  ```yaml
  gate_evidence:
    requirements_coverage:
      path: "<repo-relative 真实文件>"
      heading: "<唯一 Markdown 标题文本>"
  ```

- validator：文件存在、位于 feature/review 允许根、标题在文件中 **恰好出现一次**；`assets` 仍只含真实 path，**禁止** `path#heading`。
- `requirements_coverage: full` 时：以独立 coverage 资产路径为准；`gate_evidence` 可省略。

### 4. check-ok stamp 携带 `outcome`

`dev-flow-feature-check --finish` 成功时写入 `.claude/runtime/dev-flow/<feature-id>.check-ok`：

```text
fingerprint: <40-char-git-hash>
outcome: verified|partial
checked_at: <ISO-8601>
```

- `finish-guard`：指纹新鲜则 **允许** commit/push/merge（partial 与 verified 相同）；不得把 partial 叙述成「验证通过」。
- 业务 diff 变化导致指纹过期：partial / verified 一律失效，需重跑 check。
- `outcome: verified` 时：`accepted_risks` 必须为空，且不得存在 partial-acceptance 文件（或文件须已删除/关闭）；有未关闭 AR → FAIL。
- `outcome: partial` 时：三方一致（手测表、`accepted_risks`、`partial-acceptance.md`）见上文；stamp 与 completion 均须为 partial。

### 5. 命令名与调度

- 编排脚本统一命名为 **`task-schedule`**（路径：`.claude/skills/executing-plans/scripts/task-schedule`）。
- 文档、技能、测试中 **不得** 再使用 `sdd-schedule` 作为命令名。
- `Writes` 重叠判定：文件路径、父目录、子目录、glob 交集均视为重叠 → 串行。

### 6. 手测结果的机器可读源

- 人读：固定七列表格（上文）。
- **机器校验优先** manual-test 文件 frontmatter（或同目录侧车 YAML）中的步骤列表，例如：

  ```yaml
  manual_test_steps:
    - id: MT-001
      result: passed|failed|skipped
      risk_id: null   # skipped 时必填 AR-xxx
      observed: "..."
      evidence: "..."
  ```

- 仅有表格、无结构化步骤时：validator 可尝试解析表格，失败则 FAIL（验收需覆盖错列/空结果）。
- 禁止把「表头含实测」且单元格为 `_待执行_` / `待执行` / 空 判为通过。

### 7. `accept-risk` 与 partial-acceptance

- 每次成功 `accept-risk`：追加/更新 `<REVIEW_ROOT>/<feature-id>-partial-acceptance.md` 中对应 `AR-xxx`。
- 无任何 `skipped` 步骤时：**不得**生成或保留有效 partial-acceptance；`repair` 不得凭空创建 AR。
- `complete-gate verification-before-completion` / `--finish`：校验手测步骤、status.`accepted_risks`、partial-acceptance 三方 ID 与 step 引用一致。

### 8. 源仓开发 vs 对外发布

- **对外**：不向目标项目分发缺少 upgrade/manifest 的中间版本；目标仓只应看到完整 `workflow_version: 0.7.0`。
- **源仓**：允许堆叠 PR/commit 实现（建议顺序：status/validate/partial → authorization/guard → 技能编排 → upgrade/文档）；全部测试绿灯后 **一次性** 将 `contract.json` 的 `workflow_version` 标为 `0.7.0` 并更新 manifest 哈希。

### 9. line budget 与文档拆分

- 允许在 `contract.json` `line_budgets` 中上调 `protocol.md` / `SKILL.md` 上限，**或**把 partial、authorization、status CLI 细节外置到：
  - `.claude/skills/dev-flow/references/status-cli.md`
  - `.claude/skills/dev-flow/references/partial-verification.md`
- 禁止为塞进预算而删掉 C1/门禁硬规则。

### 10. deny fixture 分层

| 层 | 要求 | 是否挡合并 |
|----|------|------------|
| 单元 | hook 读入 PreToolUse JSON，stdout 含 `permissionDecision: "deny"`（及 ask/allow 矩阵） | **是** |
| 可选 smoke | 真实 Claude Code 会话中 Edit 被拒 | **否**（标注 manual-only，文档说明如何跑） |

不得把「仅有真实 Claude Code e2e」作为 CI 合并前提。

### 11. grillme 名额

- 「最多五个」计的是 **阻塞决策**，不是澄清性问答。
- 非阻塞澄清（复述、确认听清、查代码后的事实核对）**不占名额**。
- 仅在架构新分支 / 不可逆 / 配额成本等触发时扩到八个阻塞决策。

---

## 非目标

- 不升级 status schema 至 v4（v3 可增可选字段，见补钉 §3）。
- 不增加 L profile。
- 不把 Bash/任意解释器写入拦截宣传为安全边界。
- 不在 trace 中记录不可靠 token 数据。
- 不向目标项目安装任何未包含 upgrade 与 manifest 的中间版本。
- 不支持同一 worktree 并行两个进行中的 write-authorization feature（见补钉 §2）。
